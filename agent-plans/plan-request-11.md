## 📋 Product Specification

# Auth Microservice — Product Specification

## 1. Summary
We will build a production-ready authentication microservice in Node.js with TypeScript. The service will handle user registration, login, token refresh, logout, and provide a protected endpoint for retrieving current user info. The system will use Express.js, JWTs, bcrypt for password hashing, in-memory refresh token storage, zod for validation, and enforce strict TypeScript and unit tests for all endpoints.

## 2. User Stories

### Story 1: User Registration
**As a** new user  
**I want to** sign up using my email and password  
**So that** I can securely create an account

**Acceptance Criteria:**
- POST `/auth/signup` accepts valid email and password
- Password is hashed using bcrypt (12 rounds)
- Duplicate email results in a clear error (409 Conflict)
- Returns appropriate HTTP status and success message

---

### Story 2: User Login & Token Issuance
**As a** registered user  
**I want to** log in with my credentials  
**So that** I receive tokens for authenticated access

**Acceptance Criteria:**
- POST `/auth/login` accepts email and password
- Verifies credentials and issues JWT access token (15min) and refresh token (7 days)
- Invalid credentials return 401 Unauthorized with error message

---

### Story 3: Token Management & Protected Route
**As a** logged-in user  
**I want to** refresh my tokens or log out and access my profile  
**So that** I can maintain or revoke my session and view my info securely

**Acceptance Criteria:**
- POST `/auth/refresh` validates refresh token, issues new access token
- POST `/auth/logout` invalidates the refresh token
- GET `/auth/me` returns current user info when valid JWT is provided

---

## 3. Scope

**In Scope:**
- All endpoints (`/auth/signup`, `/auth/login`, `/auth/refresh`, `/auth/logout`, `/auth/me`)
- Password hashing (bcrypt, 12 rounds)
- JWT issuance/validation (with correct expiry)
- In-memory refresh token management
- Input validation using zod
- Clear HTTP codes and error handling
- TypeScript (strict)
- Unit testing of all endpoints with Jest

**Out of Scope:**
- Persistent user storage (e.g., database integration)
- Third-party OAuth or social authentication
- Password reset or email verification flows
- Deployment, monitoring, or scaling concerns

## 4. Estimated Complexity

**Medium** — The microservice uses standard patterns for authentication but requires careful handling of token logic, secure password storage, strict type safety, and comprehensive unit testing for all endpoints. No persistent storage or external integrations are required, keeping infrastructure complexity moderate.

## 🏗️ Architecture Design

# Auth Microservice — Technical Architecture

---

## 1. Overview

A standalone Express.js microservice built in TypeScript strict mode that handles the full authentication lifecycle: registration, login, token refresh, logout, and identity verification. JWTs are issued for stateless access control, while refresh tokens are tracked server-side in an in-memory `Map` to allow explicit revocation. All endpoints are validated with Zod schemas and covered by Jest unit tests using `supertest`.

## 🔎 Review Feedback

# Code Review: Auth Microservice

---

## 1. Overall Score

**7.5 / 10**

The plan is well-structured, follows established patterns, and covers the core authentication lifecycle competently. It earns strong marks for separation of concerns, TypeScript strictness, and test coverage intent. However, several security gaps — particularly around the in-memory store design, token handling, and missing hardening measures — prevent a higher score. These are addressable without rearchitecting the service.

---

## 2. Verdict

> **needs_revision**

The foundation is solid enough to build on, but the security concerns below should be resolved before a developer writes the first line of implementation code. None of the issues require a redesign — they require additions and corrections to the spec.

---

## 3. Strengths

- **Clean separation of concerns.** `AuthService`, `TokenService`, `UserStore`, `RefreshTokenStore`, and the controller layer are properly isolated. Business logic will not leak into HTTP handling.
- **App factory pattern.** Exporting `createApp()` without calling `listen()` is the correct approach for supertest-based testing. Many plans get this wrong.
- **Typed error hierarchy.** `AppError` with `statusCode` and machine-readable `code` fields gives the error handler everything it needs without leaking internals.
- **Zod at the boundary.** Input validation is centralized in middleware before any service code runs. The schema definitions are appropriately strict (password min/max, email format).
- **`UserPublic` projection.** Explicitly defining a response type that excludes `passwordHash` is a good habit that prevents accidental leakage.
- **Store `clear()` methods for test isolation.** Including these in the design spec (not just hoping tests handle it) is pragmatic and correct.
- **bcrypt round count assertion in tests.** Requiring `bcrypt.getRounds(hash) === 12` as a test assertion is an excellent defensive practice rarely seen in plans at this level.
- **Refresh token as opaque random bytes.** Using `crypto.randomBytes(64)` rather than a second JWT for the refresh token is architecturally sound and avoids JWT-specific attacks on the refresh path.

## 4. Concerns

### 🔴 Critical

---

**C1 — Hardcoded Fallback JWT Secret**
- **Category:** Security / Secrets Management
- **Severity:** Critical
- **Description:** `env.ts` uses `?? 'dev-secret-change-in-production'` as a fallback for `JWT_SECRET`. In a "production-ready" service, this means a missing environment variable silently produces a valid, predictable signing key. Any token signed with the fallback can be forged by anyone who reads the source code.
- **Recommendation:** Fail fast. If `JWT_SECRET` is absent, throw at startup and refuse to boot.
  ```typescript
  if (!process.env.JWT_SECRET) {
    throw new Error('FATAL: JWT_SECRET environment variable is not set.');
  }
  export const env = {
    JWT_SECRET: process.env.JWT_SECRET,
    // ...
  };
  ```
  Additionally, enforce a minimum secret length (e.g., 32 characters) and document the requirement in `.env.example`.

---

**C2 — In-Memory Stores Are Not Scoped to the App Instance**
- **Category:** Architecture / Test Integrity
- **Severity:** Critical
- **Description:** The plan describes `UserStore` and `RefreshTokenStore` as module-level singletons (`private users = new Map()`). When `createApp()` is called multiple times in tests, all instances share the same underlying store state. The `clear()` approach in `tests/setup.ts` will work only if the singleton is imported and cleared directly — but if a test file imports the app and the store separately, import caching behavior in Node.js can cause subtle ordering bugs. More seriously, this architecture makes it impossible to run parallel test workers correctly.
- **Recommendation:** Inject store instances into services rather than importing them as singletons. Pass stores into `createApp()` or use a dependency injection pattern. This also makes mocking in unit tests trivial.
  ```typescript
  export function createApp(deps?: { userStore?: UserStore; tokenStore?: RefreshTokenStore }): Express {
    const userStore = deps?.userStore ?? new UserStore();
    const tokenStore = deps?.tokenStore ?? new RefreshTokenStore();
    // inject into services...
  }
  ```

---

### 🟠 High

---

**H1 — No Rate Limiting on Authentication Endpoints**
- **Category:** Security / Brute Force
- **Severity:** High
- **Description:** `/auth/login` and `/auth/refresh` have no rate limiting. An attacker can attempt unlimited credential stuffing or brute-force attacks. The spec explicitly calls this "production-ready."
- **Recommendation:** Add `express-rate-limit` as a dependency. Apply a strict limiter (e.g., 10 requests / 15 minutes per IP) to `/auth/login` and `/auth/refresh`. Apply a more lenient limiter to `/auth/signup` to prevent account enumeration spam.
  ```typescript
  import rateLimit from 'express-rate-limit';
  const loginLimiter = rateLimit({ windowMs: 15 * 60 * 1000, max: 10 });
  router.post('/login', loginLimiter, validate(LoginSchema), AuthController.login);
  ```

---

**H2 — Logout Does Not Require Authentication**
- **Category:** Security / Authorization
- **Severity:** High
- **Description:** `POST /auth/logout` is listed as not requiring auth (`❌` in the endpoint table). This means any party who obtains a refresh token string — even without knowing the corresponding access token — can silently revoke another user's session. More critically, the current design relies solely on the client providing a valid refresh token, with no cross-validation against a known user context. A stolen refresh token can be revoked by an attacker to lock the legitimate user out.
- **Recommendation:** Either (a) require a valid Bearer JWT on `/auth/logout` and validate that the refresh token being revoked belongs to `req.user.sub`, or (b) at minimum, verify that the refresh token's `userId` matches a parameter supplied in the request. Option (a) is strongly preferred.

---

**H3 — No Refresh Token Rotation**
- **Category:** Security / Token Reuse
- **Severity:** High
- **Description:** `/auth/refresh` issues a new access token but does not rotate (invalidate + replace) the refresh token. This means a stolen refresh token remains valid for its full 7-day window after being used. The spec says "issues new access token" only — rotation is not mentioned.
- **Recommendation:** On every successful `/auth/refresh` call, revoke the presented refresh token and issue a new one. Return both `accessToken` and `refreshToken` in the response. Consider implementing refresh token families: if a previously-revoked token is presented, revoke all tokens for that user (detects token theft).

---

**H4 — `JWT_REFRESH_EXPIRES_IN` Environment Variable Is Unused**
- **Category:** Architecture / Consistency
- **Severity:** High
- **Description:** `.env.example` defines `JWT_REFRESH_EXPIRES_IN=7d`, but the `RefreshTokenRecord` model computes expiry as `now + 7 days` in TypeScript (hardcoded). The environment variable is never referenced in `TokenService`. This creates a silent misconfiguration risk: changing `JWT_REFRESH_EXPIRES_IN` in the environment has no effect, but an operator would reasonably expect it to.
- **Recommendation:** Either remove `JWT_REFRESH_EXPIRES_IN` from `.env.example` and document that refresh TTL is hardcoded, or parse and use it in `TokenService.createRefreshToken()`. Given that `JWT_ACCESS_EXPIRES_IN` is used by `jsonwebtoken` directly, apply the same pattern to refresh token TTL.

---

**H5 — Missing Security Headers**
- **Category:** Security / HTTP Hardening
- **Severity:** High
- **Description:** The app has no security headers (no `X-Content-Type-Options`, no `X-Frame-Options`, no `Content-Security-Policy`, no `Strict-Transport-Security`). For a microservice, headers like `X-Content-Type-Options: nosniff` and removing `X-Powered-By` are baseline hygiene.
- **Recommendation:** Add `helmet` as a dependency and apply it as the first middleware in `createApp()`. This is a one-line addition that addresses an entire category of HTTP-level vulnerabilities.
  ```typescript
  import helmet from 'helmet';
  app.use(helmet());
  app.disable('x-powered-by'); // redundant with helmet but explicit
  ```

---

### 🟡 Medium

---

**M1 — Password Max Length of 72 Is Correct But Undocumented — Silently Truncates**
- **Category:** Security / UX
- **Severity:** Medium
- **Description:** bcrypt silently truncates passwords at 72 bytes. The Zod schema sets `max(72)`, which is correct, but the number `72` has no comment explaining why. Future maintainers may remove or change it without understanding the security implication. Passwords with multibyte Unicode characters could also be shorter than 72 characters but longer than 72 bytes — the validation is byte-unsafe.
- **Recommendation:** Add an inline comment: `// bcrypt silently truncates at 72 bytes; enforce at schema level`. For Unicode safety, either (a) validate byte length rather than character length, or (b) pre-hash the password with SHA-256 before bcrypt (with a documented rationale about the tradeoffs).

---

**M2 — `req.user` Declaration Merging Is Not Specified**
- **Category:** TypeScript / Type Safety
- **Severity:** Medium
- **Description:** The plan mentions "attach decoded payload to `req.user` via declaration merging" but provides no type definition location or shape. This is a common source of `any` creep in Express TypeScript projects.
- **Recommendation:** Explicitly define the augmentation in the architecture:
  ```typescript
  // src/types/express.d.ts
  import { JwtPayload } from '../services/TokenService';
  declare global {
    namespace Express {
      interface Request {
        user?: JwtPayload;
      }
    }
  }
  ```
  Include this file in `tsconfig.json`'s `include` array.

---

**M3 — Email Normalization Is Underspecified**
- **Category:** Data Integrity
- **Severity:** Medium
- **Description:** The `User` model notes `email: lowercase, unique` in a comment, but there is no enforcement point specified. If `UserStore.create()` does not normalize before storage, and `findByEmail()` does not normalize before lookup, then `User@Example.com` and `user@example.com` will be treated as different users.
- **Recommendation:** Normalize email to lowercase in `AuthService.signup()` and `AuthService.login()` explicitly, before any store interaction. Add a test case for case-insensitive duplicate detection.

---

**M4 — No CORS Configuration**
- **Category:** Security / HTTP
- **Severity:** Medium
- **Description:** No CORS middleware is mentioned. If this service is consumed by a browser-based frontend (likely, given it's an auth service), absent CORS configuration means all cross-origin requests will be blocked — or, if a developer adds a wildcard `*` origin ad-hoc, it could undermine cookie-based security.
- **Recommendation:** Add `cors` as a dependency. Configure it explicitly with an allowlist of origins loaded from environment variables. Document this in `.env.example`.

---

**M5 — `tests/setup.ts` Store Reset Mechanism Is Fragile**
- **Category:** Test Architecture
- **Severity:** Medium
- **Description:** The plan says `beforeEach` clears both stores, but if stores are singletons, each test file must import the same singleton instance for `clear()` to have any effect. There's no guarantee that the `createApp()` call in each test file and the `clear()` call in `setup.ts` are operating on the same object — especially if Jest's module registry is reset between files.
- **Recommendation:** This is the downstream consequence of C2. Resolving C2 (dependency injection) resolves this automatically. If singletons are kept, document explicitly that `jest.config.ts` must use `--runInBand` (serial execution) and `resetModules: false`.

---

**M6 — No Request ID / Correlation Logging**
- **Category:** Observability
- **Severity:** Medium
- **Description:** The error handler is specified to suppress stack traces (correct), but there is no mechanism to correlate a client-facing error with a server-side log entry. In production, this makes debugging nearly impossible.
- **Recommendation:** Generate a `requestId` (UUID) per request in a middleware and attach it to `req`. Include it in error log output and in the error response body. This does not require a logging library — `console.error(requestId, err)` is sufficient for this scope.

### 🟢 Low

- **Severity:** Low
- **Description:** Node.js 14.17+ includes `crypto.randomUUID()` natively. Adding `uuid` as a runtime dependency for this single use case adds unnecessary package weight and a potential supply chain surface.
- **Recommendation:** Replace `import { v4 as uuidv4 } from 'uuid'` with `crypto.randomUUID()` and remove the `uuid` and `@types/uuid` dependencies.

---

**L2 — `ts-node` in Production Path**
- **Category:** Build / Operations
- **Severity:** Low
- **Description:** The plan installs `ts-node` as a dev dependency but does not specify whether `src/index.ts` is run via `ts-node` or compiled `dist/` in production. If `ts-node` is used at runtime, startup will be significantly slower and memory usage higher.
- **Recommendation:** Add a `build` script (`tsc`) and a `start` script that runs `node dist/index.js`. Use `ts-node` only in development (`dev` script). Document this in `package.json` scripts section.

---

**L3 — Logout Response on Invalid Token Should Be Idempotent**
- **Category:** API Design
- **Severity:** Low
- **Description:** The test plan lists "already-revoked token" as a case for logout, but the HTTP status code mapping doesn't specify what to return for that case. Returning `401` on a second logout attempt leaks information about session state and can frustrate legitimate clients doing defensive double-logout.
- **Recommendation:** `POST /auth/logout` should return `200` regardless of whether the token was already revoked or never existed. Logout is inherently idempotent — the goal is "this token is not valid," and after any logout attempt, that goal is achieved.
