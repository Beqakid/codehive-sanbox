# Product Specification: PCS App - Caregiver Marketplace

## 1. Summary
The PCS App is a mobile-friendly caregiver marketplace built using the Hono web framework, deployed on Cloudflare. The platform will feature a public landing page, as well as two separate Progressive Web Apps (PWAs): one for caregivers to manage profiles and connect with clients, and another for clients to find, book, and review caregivers.

---

## 2. User Stories

### a. As a client, I want to browse and book qualified caregivers so that I can find suitable care for my needs.
**Acceptance Criteria:**
- Clients can create accounts and log in via the client PWA.
- Clients can search and filter caregiver profiles by relevant criteria (location, expertise, availability).
- Clients can book a caregiver and view their booking history.

### b. As a caregiver, I want to create and manage my profile and availability so that I can be found and contacted by clients.
**Acceptance Criteria:**
- Caregivers can register and log in via the caregiver PWA.
- Caregivers can create/edit a profile, including uploading a photo, adding experience, and setting availability.
- Caregivers can view incoming client requests and respond.

### c. As a visitor, I want to learn about the appвЂ™s value proposition and sign up as a client or caregiver.
**Acceptance Criteria:**
- Landing page highlights the benefits of the marketplace.
- Clear calls to action for both client and caregiver sign-ups.

---

## 3. Scope

### In Scope
- Build and deploy the app using Hono + Cloudflare stack.
- Responsive landing page accessible via any web browser.
- Separate PWAs for caregiver and client roles, both mobile-first.
- Authentication and basic profile management for both user types.
- Search, listing, and booking workflows.

### Out of Scope
- In-app payment processing or advanced scheduling logic.
- Detailed notification system (beyond MVP email or in-app messages).
- Native mobile (iOS/Android) appsвЂ”PWAs only.
- Advanced administrative features (e.g., analytics dashboard, role management).

---

## 4. Estimated Complexity

**Medium.**  
Rationale: The project requires multi-role authentication, PWA implementation, and integration with the Hono framework on Cloudflare, but avoids complex real-time or payment features in the MVP. Building mobile-friendly, role-specific apps introduces moderate complexity but is manageable with existing web technologies.

# PCS App вЂ” Technical Architecture

## 1. Overview

The PCS App is a monorepo built on Cloudflare's edge infrastructure using Hono as the API layer, with two distinct React-based PWAs (client and caregiver) served from Cloudflare Pages and backed by a Hono API deployed to Cloudflare Workers. Authentication is handled via JWT stored in secure HTTP-only cookies, with Cloudflare D1 (SQLite) as the primary database and Cloudflare R2 for file/image storage.

## 4. API Endpoints

All routes prefixed with `/api/v1`. Auth routes are public; all others require `Authorization: Bearer <token>` or session cookie.

### Auth
```
POST   /api/v1/auth/register          # register as client or caregiver
POST   /api/v1/auth/login             # returns JWT + refresh token
POST   /api/v1/auth/logout
POST   /api/v1/auth/refresh           # exchange refresh token
GET    /api/v1/auth/me                # current user info
```

### Caregiver
```
GET    /api/v1/caregivers             # search/list (public) ?city=&specialty=&available=&page=
GET    /api/v1/caregivers/:id         # public profile detail
PUT    /api/v1/caregivers/profile     # update own profile [CAREGIVER]
POST   /api/v1/caregivers/avatar      # upload photo to R2 [CAREGIVER]
GET    /api/v1/caregivers/availability          # get own slots [CAREGIVER]
POST   /api/v1/caregivers/availability          # add slot [CAREGIVER]
DELETE /api/v1/caregivers/availability/:id      # remove slot [CAREGIVER]
```

### Client
```
GET    /api/v1/clients/profile        # get own profile [CLIENT]
PUT    /api/v1/clients/profile        # update own profile [CLIENT]
```

### Bookings
```
POST   /api/v1/bookings               # create booking request [CLIENT]
GET    /api/v1/bookings               # list own bookings (both roles)
GET    /api/v1/bookings/:id
PATCH  /api/v1/bookings/:id/status    # accept/decline/cancel [CAREGIVER or CLIENT]
```

### Reviews
```
POST   /api/v1/reviews                # submit review after completed booking [CLIENT]
GET    /api/v1/reviews/caregiver/:id  # public reviews for a caregiver
```

## 4. Concerns

### рџ”ґ CRITICAL

**C-1 | Security | No authorization layer вЂ” IDOR vulnerabilities guaranteed**

**Category:** Authorization  
**Severity:** Critical

The API design lists role-based guards (`[CAREGIVER]`, `[CLIENT]`) but there is **no mention of object-level authorization anywhere**. Any authenticated user can currently:

- `GET /api/v1/bookings/:id` вЂ” fetch any booking by ID
- `PATCH /api/v1/bookings/:id/status` вЂ” attempt to accept/cancel any booking
- `GET /api/v1/caregivers/availability` вЂ” ambiguous; which caregiver?

This is a textbook **Insecure Direct Object Reference (IDOR)** vulnerability. On a platform with care recipient personal data, this is a critical data breach risk.

**Recommendation:**
```typescript
// In every resource handler, enforce ownership:
// bookings.ts
app.get('/bookings/:id', authMiddleware, async (c) => {
  const booking = await db.getBooking(c.req.param('id'));
  const user = c.get('user');
  
  // MUST check ownership before returning data
  if (booking.client_id !== user.id && booking.caregiver_id !== user.id) {
    return c.json({ error: 'Forbidden' }, 403);
  }
  return c.json(booking);
});
```

Add an explicit authorization checklist to every route as a required code review gate.

**C-2 | Security | Role enforcement is client-controlled at registration**

**Category:** Authentication / Authorization  
**Severity:** Critical

```
POST /api/v1/auth/register   # register as client or caregiver
```

The `role` field is passed by the client at registration time. There is no secondary verification. This means:

1. Any user can self-assign the `caregiver` role
2. A bad actor could register as a caregiver and immediately appear in search results
3. There is no concept of background check, identity verification, or admin approval

**Recommendation:**
- Even for MVP, caregivers should require **email verification** before their profile becomes publicly discoverable
- Add `is_verified` flag gating on the `GET /api/v1/caregivers` public search (only return `is_verified: true` caregivers)
- Add a `registration_status: 'pending_verification' | 'active' | 'suspended'` field
- Document that background check integration is a Phase 2 requirement вЂ” the data model must support it now

**C-3 | Security | File upload validation is completely absent**

**Category:** Input Validation / Storage Security  
**Severity:** Critical

`POST /api/v1/caregivers/avatar` and the "direct to R2 via signed URL" approach in Step 4.3 have **no described validation**:

- No file type validation (MIME type + magic bytes)
- No file size limits defined
- No mention of image processing/stripping EXIF data (which can contain GPS coordinates вЂ” a serious privacy concern for caregivers)
- Signed URL approach means the API never sees the bytes, making server-side validation impossible

**Recommendation:**
```typescript
// Server-side validation before issuing signed URL
const MAX_AVATAR_SIZE = 5 * 1024 * 1024; // 5MB
const ALLOWED_TYPES = ['image/jpeg', 'image/png', 'image/webp'];

// Validate Content-Type and Content-Length in the presign request
// Process uploads through a Worker that:
// 1. Validates magic bytes (not just MIME header)
// 2. Strips EXIF metadata before storing to R2
// 3. Enforces max dimensions (e.g., 2000x2000)
// Consider: sharp via WASM for image processing in Workers
```

Do not issue signed URLs without server-side post-upload validation hooks.

**C-4 | Security | JWT secret management is underspecified**

**Category:** Secrets Management  
**Severity:** High

Step 6.3 mentions `wrangler secrets for JWT_SECRET` but the architecture doesn't define:

- JWT expiry durations (access token vs. refresh token)
- Refresh token storage strategy (KV? D1? In-memory only?)
- Refresh token rotation policy
- Token revocation mechanism (critical for logout and compromised account scenarios)

**Recommendation:**
```
Access token:  15-minute expiry, stateless JWT
Refresh token: 7-day expiry, stored in D1 with:
  - token_hash (not the raw token)
  - user_id
  - expires_at
  - revoked_at (nullable)
  - created_ip

On logout: set revoked_at = NOW()
On /auth/refresh: verify token not revoked before issuing new pair
```

**C-5 | Data Model | `specialties`, `certifications`, `languages` stored as raw JSON strings**

**Category:** Data Integrity / Query Performance  
**Severity:** High

```typescript
specialties: string;  // JSON array: ['elderly','dementia','pediatric']
```

Storing filterable arrays as JSON strings in D1 (SQLite) means:

- Search via `LIKE '%elderly%'` вЂ” fragile and non-indexable
- No referential integrity for valid specialty values
- Step 2.5 acknowledges "JSON contains" filtering but this will be slow and inaccurate at scale
- A specialty named `"pediatric"` would false-match a search for `"ric"`

**Recommendation:**

For SQLite/D1, use a junction table pattern:
```sql
CREATE TABLE caregiver_specialties (
  caregiver_id TEXT NOT NULL REFERENCES caregiver_profiles(id),
  specialty TEXT NOT NULL CHECK(specialty IN ('elderly','dementia','pediatric','disability','palliative')),
  PRIMARY KEY (caregiver_id, specialty)
);
CREATE INDEX idx_caregiver_specialties_specialty ON caregiver_specialties(specialty);
```

This makes `WHERE specialty = ?` indexed, accurate, and type-safe.

**C-6 | Architecture | `rating_avg` and `review_count` are denormalized with no sync strategy**

**Category:** Data Integrity  
**Severity:** High

```typescript
rating_avg: number;  // computed, stored for perf
review_count: number;
```

The architecture notes these are "stored for perf" but defines no mechanism to keep them consistent. If a review is inserted and the update to `caregiver_profiles` fails, or if a review is ever deleted, these values will be permanently wrong. D1 does not support stored procedures or triggers.

**Recommendation:**

Use a D1 transaction to update both atomically on every review write:
```typescript
// In reviewService.ts
await db.batch([
  db.prepare('INSERT INTO reviews ...').bind(...),
  db.prepare(`
    UPDATE caregiver_profiles 
    SET review_count = review_count + 1,
        rating_avg = (rating_avg * review_count + ?) / (review_count + 1)
    WHERE id = ?
  `).bind(newRating, caregiverId)
]);
```

Or alternatively, compute `rating_avg` at query time with a JOIN and cache the result in KV with a short TTL. Prefer correctness over performance at MVP scale.

**C-7 | Security | PII on care recipients has no access controls documented**

**Category:** Privacy / Data Protection  
**Severity:** High

```typescript
interface ClientProfile {
```

This is sensitive data about a third party (the care recipient) who is not a user of the system and cannot consent through the app flow. The architecture has no mention of:

**C-9 | Architecture | Search endpoint will not scale with D1's query limitations**

**Category:** Performance  
**Severity:** Medium

The search endpoint (`GET /api/v1/caregivers`) needs to filter by city, specialty, availability, and rating, with pagination. D1 (SQLite) has no full-text search, no geospatial indexing, and no vector search. At even modest scale (1,000+ caregivers), a multi-filter query joining across tables will be slow.

**Recommendation:**

For MVP, this is acceptable with proper indexing:
```sql
CREATE INDEX idx_caregiver_location ON caregiver_profiles(location_state, location_city);
CREATE INDEX idx_caregiver_available ON caregiver_profiles(is_available, rating_avg DESC);
```

But add a **Phase 2 ticket** explicitly for migrating search to Cloudflare Vectorize or an external search service (Meilisearch, Typesense, or Algolia). The current D1 approach should be treated as a known technical debt item, not a long-term solution.

**C-10 | Architecture | Refresh token endpoint is potentially exposed to CSRF**

**Category:** Security  
**Severity:** Medium

`POST /api/v1/auth/refresh` uses a cookie-based refresh token. If CORS is misconfigured (which it often is in early development), this endpoint is vulnerable to cross-site request forgery.

**Recommendation:**
- Set `SameSite=Strict` on the refresh token cookie
- Verify the `Origin` header on the refresh endpoint
- Add CSRF token validation for any state-changing cookie-based endpoints

**C-11 | UX/Security | Both PWAs share the same `/api/v1/auth/register` endpoint with role selection**

**Category:** Security / UX  
**Severity:** Medium

Nothing technically prevents a user of the caregiver PWA from registering with `role=client` or vice versa. While this might seem minor, it creates confusion and potential policy violations.

**Recommendation:**

Have the caregiver PWA hardcode `role: 'caregiver'` on the client side and have the server enforce that the `role` matches the expected value for the endpoint context. Alternatively, use separate registration endpoints: `/auth/register/caregiver` and `/auth/register/client` where the role is set server-side, not client-supplied.

**Category:** API Design

```typescript
// packages/types/src/errors.ts
```

**C-13 | Architecture | Booking conflict detection is not addressed**

**Category:** Business Logic  
**Severity:** Medium

A caregiver can receive multiple overlapping booking requests. The current booking model and API have no mechanism to:

- Detect time conflicts before accepting
- Prevent double-booking after acceptance
- Notify the client if a booking is declined due to conflict vs. other reasons

**Recommendation:**

Add a conflict check query in the booking acceptance handler:
```sql
SELECT COUNT(*) FROM bookings 
WHERE caregiver_id = ? 
  AND status = 'accepted'
  AND scheduled_date = ?
  AND NOT (end_time <= ? OR start_time >= ?)
```

Run this within the status update transaction and reject acceptance if conflicts exist.

**C-14 | Architecture | `bcryptjs WASM build` is a runtime risk in CF Workers**

**Category:** Dependency Risk  
**Severity:** Low

Workers have a 1MB compressed script size limit and strict CPU time limits (10ms on free tier, 30ms on paid). bcrypt is intentionally CPU-intensive. A single bcrypt operation with cost factor 12 takes ~300ms, which **will exceed the Workers CPU limit**.

**Recommendation:**

const hash = await crypto.subtle.deriveBits({ name: 'PBKDF2', salt, iterations: 600_000, hash: 'SHA-256' }, key
