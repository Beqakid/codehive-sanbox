## 📋 Product Specification

# Product Specification: User Authentication for CodeHive Sandbox

## Summary
The CodeHive Sandbox project will implement a secure user authentication system using JSON Web Tokens (JWT) stored in **httpOnly cookies**, with a **server-side token revocation list**, **rate limiting** on the login endpoint, and **robust middleware** that validates token expiration and claims. A protected dashboard route allows only authenticated users access.

## User Stories

### User Story 1: User Login
**As** a user,  
**I want** to be able to log into my account using my credentials,  
**So that** I can access my personalized dashboard.

**Acceptance Criteria:**
- Users enter their username and password to log in.
- Upon successful authentication, a short-lived JWT (15 min) is issued as an httpOnly, Secure, SameSite=Strict cookie.
- A refresh token (7 days) is issued separately as an httpOnly cookie.
- Users receive a generic error message for incorrect credentials (no info leakage).
- Login endpoint enforces rate limiting: max 5 attempts per IP per minute.

### User Story 2: User Logout
**As** a user,  
**I want** to be able to log out of my account,  
**So that** I can secure my session.

**Acceptance Criteria:**
- Users can click a "Logout" button.
- The server adds the current JWT's `jti` (JWT ID) to a revocation list in D1.
- The server clears both the access token and refresh token cookies.
- Users are redirected to the login page after logout.

### User Story 3: Protected Dashboard Access
**As** a user,  
**I want** to access the dashboard only when I'm logged in,  
**So that** my information is secured from unauthorized access.

**Acceptance Criteria:**
- Unauthenticated users are redirected to the login page.
- Middleware validates: token signature, expiration (`exp`), issued-at (`iat`), subject (`sub`), and checks `jti` against the revocation list.
- Access is granted only when all checks pass.

## Scope
### In-Scope
- JWT login/logout with httpOnly cookies.
- Access token (15 min) + refresh token (7 days) pattern.
- Server-side token revocation list (D1 `revoked_tokens` table).
- Rate limiting on `/api/login` (KV or in-memory counter).
- Protected dashboard route with hardened middleware.
- Comprehensive error handling that does not leak sensitive info.
- Failed login attempt logging.

### Out of Scope
- User registration.
- Profile management.
- Password recovery.
- OAuth2 / OpenID Connect (future consideration).

## Estimated Complexity
**Medium-High**  
The security hardening (httpOnly cookies, revocation list, rate limiting, claims validation) adds meaningful complexity over a basic JWT implementation but is the right choice for production.

## 🏗️ Architecture Design

# Technical Architecture for User Authentication in CodeHive Sandbox

## Overview
Authentication uses short-lived JWTs (HS256 with a 256-bit secret stored as a Worker secret) delivered via **httpOnly cookies** — never `localStorage`. A **D1-backed revocation list** ensures tokens are invalidated server-side on logout. The login endpoint is protected by **rate limiting** using Cloudflare KV. All middleware performs **full claims validation** including expiration and `jti` revocation checks.

## Security Decisions

| Concern | Solution |
|---|---|
| XSS token theft | httpOnly, Secure, SameSite=Strict cookies — JS cannot access tokens |
| Token reuse after logout | `jti` stored in D1 `revoked_tokens` on logout; middleware rejects revoked `jti` |
| Brute force on login | Rate limit: 5 attempts / IP / 60s via Cloudflare KV |
| Weak token validation | Middleware checks: signature, `exp`, `iat`, `sub`, `jti` revocation |
| Weak password storage | bcrypt with cost factor 12 + unique salt per user |
| Info leakage on errors | Generic "Invalid credentials" message for all auth failures |

## API Endpoints

### `POST /api/login`
- **Rate limit:** 5 req / IP / 60s (429 if exceeded).
- **Request:** `{ username: string, password: string }`
- **On success:** Set `access_token` cookie (httpOnly, Secure, SameSite=Strict, maxAge=900) + `refresh_token` cookie (httpOnly, Secure, SameSite=Strict, maxAge=604800). Response: `{ ok: true }`.
- **On failure:** `{ error: 'Invalid credentials' }` (same message regardless of whether user exists).
- **Logging:** Log failed attempts with IP and timestamp (no passwords).

### `POST /api/logout`
- Read `access_token` cookie → decode `jti` → insert into D1 `revoked_tokens`.
- Clear both cookies (maxAge=0).
- Response: `{ ok: true }`.

### `POST /api/refresh`
- Read `refresh_token` cookie → verify → issue new `access_token` cookie.
- If refresh token invalid/expired: 401, clear cookies.

### `GET /api/me`
- Protected by `requireAuth` middleware.
- Response: `{ id, username }` from token claims.

### `GET /api/dashboard`
- Protected by `requireAuth` middleware.
- Response: user dashboard data.

## Middleware: `requireAuth`

```typescript
// utils/middleware.ts
export async function requireAuth(req, res, db) {
  const token = req.cookies['access_token'];
  if (!token) return { authed: false, reason: 'no_token' };

  const payload = await verifyToken(token); // throws if sig invalid
  if (!payload) return { authed: false, reason: 'invalid_token' };

  // 1. Check expiration
  if (Date.now() / 1000 > payload.exp)
    return { authed: false, reason: 'expired' };

  // 2. Check issued-at (reject tokens issued in the future)
  if (payload.iat > Date.now() / 1000 + 30)
    return { authed: false, reason: 'invalid_iat' };

  // 3. Check subject claim
  if (!payload.sub)
    return { authed: false, reason: 'missing_sub' };

  // 4. Check revocation list
  const revoked = await db.prepare(
    'SELECT 1 FROM revoked_tokens WHERE jti = ?'
  ).bind(payload.jti).first();
  if (revoked) return { authed: false, reason: 'revoked' };

  return { authed: true, userId: payload.sub };
}
```

## Rate Limiting: `checkRateLimit`

```typescript
// utils/rateLimit.ts
export async function checkRateLimit(ip: string, kv: KVNamespace): Promise<boolean> {
  const key = `login_attempts:${ip}`;
  const current = parseInt(await kv.get(key) ?? '0');
  if (current >= 5) return false; // blocked
  await kv.put(key, String(current + 1), { expirationTtl: 60 });
  return true;
}
```

3. **Worker Secrets** — Add `JWT_SECRET` (256-bit random) and `JWT_REFRESH_SECRET` as Worker secrets.
4. **Token Service** — Implement `createAccessToken` (15 min, includes `jti`), `createRefreshToken` (7 days), `verifyToken` (full claims check).
5. **Rate Limiter** — KV sliding window for `/api/login`.
6. **Login Endpoint** — Rate check → bcrypt verify → issue httpOnly cookies.
7. **Logout Endpoint** — Insert `jti` into D1 → clear cookies.
8. **Refresh Endpoint** — Verify refresh token → issue new access token cookie.
9. **Middleware** — Full validation: sig + exp + iat + sub + jti revocation.
10. **Protected Routes** — Apply `requireAuth` to `/api/me`, `/api/dashboard`.
11. **Frontend** — Login form (no token handling — cookies are automatic), AuthProvider via `/api/me`, LogoutButton.
12. **Logging** — Log auth events (no sensitive data).
13. **Testing** — Happy path, invalid credentials, expired token, revoked token, rate limit trigger, refresh flow.
14. **Documentation** — Update README with setup, secrets needed, migration instructions.
