# Product Specification: Auth Microservice

## 1. Summary
We will build a production-ready authentication microservice using Node.js and TypeScript. The service will issue JWT access tokens and manage refresh token rotation while providing protected routes for secure resource access. Security best practices such as password hashing and route protection will be strictly followed.

## 2. User Stories

### User Story 1: User Registration and Login  
**As** a new or returning user,  
**I want** to register and log in with my email and password,  
**So that** I can obtain valid JWT tokens to access protected resources.

**Acceptance Criteria:**
- Users can register with a unique email/password (passwords hashed securely).
- Users can log in with valid credentials.
- On successful login, API returns both access and refresh tokens.

---

### User Story 2: Token Refresh  
**As** an authenticated user,  
**I want** to refresh my access token using a valid refresh token,  
**So that** I can maintain a secure session without repeatedly logging in.

**Acceptance Criteria:**
- Endpoint accepts a valid refresh token and issues a new access token (and rotates refresh token).
- Invalid/expired or reused refresh tokens are rejected with appropriate errors.
- Rotated refresh tokens are stored and cannot be reused.

---

### User Story 3: Protected Routes  
**As** an API consumer,  
**I want** certain endpoints to be accessible only if I provide a valid JWT,  
**So that** unauthorized requests are blocked from sensitive actions.

**Acceptance Criteria:**
- Protected endpoints require a valid access token.
- Requests with missing or invalid access tokens are rejected.
- Successful requests with valid tokens gain access to protected resources.

## 🏗️ Architecture Design

# Technical Architecture: Auth Microservice

## 1. Overview

This microservice is a self-contained Express.js/TypeScript REST API that handles user registration, login, and JWT-based session management with refresh token rotation. Access tokens are short-lived JWTs verified on each request, while refresh tokens are stored in a SQLite database (via Prisma) and rotated on every use to prevent replay attacks. All sensitive routes are gated behind a reusable JWT middleware.

## 3. Data Models

### `User`
```prisma
model User {
  id           String         @id @default(uuid())
  email        String         @unique
  passwordHash String
  createdAt    DateTime       @default(now())
  updatedAt    DateTime       @updatedAt
  refreshTokens RefreshToken[]
}
```

### `RefreshToken`
```prisma
model RefreshToken {
  id        String   @id @default(uuid())
  token     String   @unique          // hashed SHA-256 of the actual token
  userId    String
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  family    String                    // rotation family ID — detects reuse attacks
  isRevoked Boolean  @default(false)
  expiresAt DateTime
  createdAt DateTime @default(now())

  @@index([userId])
  @@index([family])
}
```

> **Rotation family**: Every login starts a new `family` UUID. Each rotation replaces the token within the same family. If a revoked token in a family is presented, **all tokens in that family are invalidated** (reuse attack detected).

```typescript
export const authenticate = async (req: Request, res: Response, next: NextFunction) => {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) {
    return next(new AppError('MISSING_TOKEN', 'Authorization token required.', 401));
  }
  const token = authHeader.split(' ')[1];
  const payload = tokenService.verifyAccessToken(token); // throws on invalid
  req.user = { id: payload.sub };
  next();
};
```

## 3. Strengths

### ✅ Refresh Token Family-Based Reuse Detection
The rotation family pattern (login → new UUID family, reuse of revoked token → wipe entire family) is the correct, modern approach to refresh token replay attacks. Many "production" auth implementations skip this entirely.

### ✅ Hashing Refresh Tokens at Rest
Storing SHA-256 of the raw token rather than plaintext is the right call. Even if the database is compromised, raw tokens are not exposed.

### ✅ Zod-Validated Environment at Startup
Failing fast on missing/malformed secrets (enforcing 32-char minimum) prevents misconfigured deployments from running silently broken. This is frequently overlooked.

### ✅ Layered Architecture with Clear Separation
Controller → Service → Repository is correctly enforced. TokenService is cleanly isolated from AuthService. This makes unit testing viable and dependencies explicit.

### ✅ Centralized Error Handling
The `AppError` class with structured error codes prevents error-format inconsistencies across endpoints and makes client-side error handling predictable.

### ✅ Cascading Deletes on User Removal
`onDelete: Cascade` on `RefreshToken.userId` prevents orphaned token records — a small but real operational correctness detail.

## 4. Concerns

### 🔴 CRITICAL — No Rate Limiting on Auth Endpoints

**Category:** Security  
**Severity:** Critical

The plan has zero mention of rate limiting on `/register`, `/login`, or `/refresh`. These are the highest-value brute-force targets in the entire service.

**Specific risks:**
- `/login` — unlimited password guessing
- `/register` — account enumeration and spam account creation
- `/refresh` — token flooding

**Recommendation:**  
Add `express-rate-limit` (or an Nginx/gateway layer) before any other middleware. Configure it with per-IP limits and sliding windows:

```typescript
import rateLimit from 'express-rate-limit';

export const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 10,                   // 10 attempts per window per IP
  standardHeaders: true,
  legacyHeaders: false,
  message: { error: { code: 'RATE_LIMITED', message: 'Too many attempts.', statusCode: 429 } }
});

// Apply to login and register only — refresh can have a separate, higher limit
app.use('/api/auth/login', authLimiter);
app.use('/api/auth/register', authLimiter);
```

### 🔴 CRITICAL — Refresh Token Transport Security Unspecified

**Category:** Security  
**Severity:** Critical

The spec sends the raw refresh token in the response body (`200 { accessToken, refreshToken }`), and the `/refresh` endpoint reads it from `req.body`. This is not inherently wrong, but it is dangerous if not paired with an explicit `HttpOnly` cookie strategy or documented client-side storage guidance.

Storing a refresh token in `localStorage` is a well-known XSS vector. The plan says nothing about this.

**Recommendation:**  
Make an explicit architectural decision. Two options:

**Option A — HttpOnly Cookie (recommended for browser clients):**
```typescript
res.cookie('refreshToken', rawToken, {
  httpOnly: true,
  secure: true,            // HTTPS only
  sameSite: 'strict',
  maxAge: env.REFRESH_TOKEN_TTL_MS,
  path: '/api/auth',       // Scope cookie to auth routes only
});
// Don't include refreshToken in body at all
```

**Option B — Body transport (acceptable for M2M/mobile):**  
Document explicitly that clients MUST store in secure storage (Keychain, EncryptedSharedPreferences) and NEVER in browser localStorage. Add this to the README as a required client contract.

### 🔴 CRITICAL — Timing Attack Vulnerability in Login Flow

**Category:** Security  
**Severity:** Critical

The login flow as specified is:
```
1. findByEmail → 404 if not found
2. bcrypt.compare → 401 if mismatch
```

Step 1 returns almost immediately when the user doesn't exist. Step 2 runs a full bcrypt comparison (~100ms) when they do. This timing difference allows an attacker to enumerate which emails are registered.

**Recommendation:**  
Always run `bcrypt.compare` regardless of whether the user is found, using a dummy hash for the non-existent user case:

```typescript
const DUMMY_HASH = await bcrypt.hash('dummy-prevent-timing', 12); // computed once at startup

const user = await userRepository.findByEmail(email);
const passwordToCompare = user?.passwordHash ?? DUMMY_HASH;
const isValid = await bcrypt.compare(password, passwordToCompare);

if (!user || !isValid) {
  throw new AppError('INVALID_CREDENTIALS', 'Email or password is incorrect.', 401);
}
```

### 🟠 HIGH — Refresh Token Expiry Check Happens in Application Code Only

**Category:** Security  
**Severity:** High

The `rotateRefreshToken` logic checks `expiresAt` in application code, but the expired token record still exists in the database indefinitely. The `deleteExpired()` method is mentioned in the repository but there is no specification of when or how it runs.

**Risks:**
- Database bloat in high-traffic scenarios
- No guarantee cleanup ever runs without an explicit schedule

**Recommendation:**  
Specify cleanup mechanism explicitly. Choose one:

```typescript
// Option A: Cron job (e.g., node-cron)
import cron from 'node-cron';
cron.schedule('0 3 * * *', () => tokenRepository.deleteExpired()); // Run at 3am daily

// Option B: Probabilistic cleanup on each refresh (lightweight, no scheduler needed)
// In rotateRefreshToken(), after issuing new token:
if (Math.random() < 0.01) { // 1% of requests trigger cleanup
  tokenRepository.deleteExpired().catch(err => logger.error(err, 'Token cleanup failed'));
}
```

Also consider adding a DB-level index on `expiresAt` if the table grows large.

**Severity:** Medium


Add `ALLOWED_ORIGINS` to the `.env.example` and Zod schema.

### 🟡 MEDIUM — JWT Payload Lacks `jti` Claim

**Category:** Security  
**Severity:** Medium

The access token payload is implied to contain only `sub` (userId). Without a `jti` (JWT ID), there is no mechanism to revoke individual access tokens before they expire (e.g., for a forced logout or security incident response).

The current logout endpoint only revokes the refresh token. An attacker who already captured the access token can continue using it for up to 15 minutes.

**Recommendation:**  
This is an accepted tradeoff for stateless JWT architectures, but it must be explicitly documented as a known limitation. If immediate revocation is required, add:

```typescript
// Option: Redis-backed token blocklist for access tokens
// On logout: store jti in Redis with TTL matching token remaining lifetime
// In auth middleware: check blocklist before accepting token
```

If Redis is out of scope, reduce `ACCESS_TOKEN_TTL` to 5 minutes and document the 5-minute revocation lag as the security boundary.

### 🟡 MEDIUM — bcrypt Cost Factor Not Validated Against Environment

**Category:** Security / Performance  
**Severity:** Medium

bcrypt cost factor 12 is a reasonable default, but it's hardcoded. In test environments, this makes tests that call `bcrypt.hash` slow (100-200ms per hash). More importantly, as hardware improves, the cost factor should be configurable without a code change.

**Recommendation:**  
```typescript
// In env.ts
BCRYPT_ROUNDS: z.coerce.number().min(10).max(14).default(12),

// In tests, set BCRYPT_ROUNDS=4 for speed without compromising test coverage validity
```

### 🟡 MEDIUM — SQLite → PostgreSQL Migration Fragility

**Category:** Architecture  
**Severity:** Medium

The architecture lists "SQLite (dev) / PostgreSQL (prod)" as a database strategy. This is a classic footgun. SQLite and PostgreSQL have meaningful behavioral differences:
- Case sensitivity in string comparisons
- Boolean representation
- UUID handling
- Concurrent write behavior (SQLite serializes writes)
- `LIKE` operator behavior differences

Email lookups that are case-insensitive on SQLite may behave differently on PostgreSQL.

**Recommendation:**  
- Use PostgreSQL via Docker (`docker-compose.yml`) even in local development. The `docker-compose.yml` is already in the file structure — use it as the primary dev database.
- If SQLite must be supported, add an explicit test matrix that runs integration tests against both databases in CI.
- Add `@@map` annotations and ensure email is stored lowercase on write:

```typescript
// In AuthService.register and login:
const normalizedEmail = email.toLowerCase().trim();
```

**Category:** Observability  
**Severity:** Low

Without request correlation IDs, tracing a failed token refresh through logs across multiple log lines is painful in production.

**Recommendation:**

```typescript
app.use((req, res, next) => {

  next();
});
```

Pass `req.id` through all log statements.

## 5. Security Notes

### Authentication
- The access token lifetime of 15 minutes is acceptable, but with no revocation mechanism it represents a fixed security lag window. Document this explicitly.
- The `signRefreshToken()` using `crypto.randomBytes(64)` is cryptographically sound — this is the right approach vs. signing a JWT as the refresh token (which would complicate revocation).
- bcrypt cost factor 12 is appropriate for current hardware. Reassess annually.

### Authorization
- The `/api/user/me` route is the only protected route in scope. Ensure the `authenticate` middleware checks `req.user.id` actively — don't attach a stale payload.
- The current design has no role/permission concept, which is fine per scope. However, the JWT payload should be kept minimal now to avoid breaking changes when roles are added later. Define `JwtPayload` narrowly:
  ```typescript
  interface JwtPayload {
    sub: string;  // userId only — add roles here later without payload bloat
    iat: number;
    exp: number;
  }
  ```

### Data Protection
- Refresh tokens are hashed (SHA-256) at rest ✅
- Passwords are bcrypt-hashed ✅
- Raw tokens must never be logged — add this explicitly to the logger configuration
- The `email` field should be normalized to lowercase before storage and lookup to prevent duplicate account creation via `User@Example.com` vs `user@example.com`
- Consider adding a `lastLoginAt` field to the User model for audit purposes without adding significant complexity

## 6. Scalability Notes

### Performance
- **SQLite in production is a non-starter** for any concurrent load. SQLite's write serialization will cause contention under concurrent login requests. The architecture acknowledges PostgreSQL for production — enforce this in deployment documentation and ideally in the `NODE_ENV=production` startup check.
- `deleteExpired()` without an index on `expiresAt` becomes a full table scan as the `RefreshToken` table grows. Add:
  ```prisma
  @@index([expiresAt])
  ```
- bcrypt hashing is CPU-intensive. Under high load, consider offloading
