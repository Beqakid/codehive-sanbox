## 📋 Product Specification

# Product Specification: User Authentication for Bookmark Manager

## 1. Summary
This feature will introduce user authentication to the Bookmark Manager using JWT (JSON Web Tokens). Users will be able to register, log in/out, and access personalized, protected routes, ensuring bookmarks and settings are securely tied to individual accounts.

## 2. User Stories

### User Story 1: Account Registration & Login
**As a** new user,  
**I want** to register for an account and log in  
**So that** I can securely access and manage my bookmarks.

**Acceptance Criteria:**
- Users can create an account with email and password.
- Users can log in with valid credentials to receive a JWT token.
- Authentication errors (e.g., wrong password) are clear and actionable.

---

### User Story 2: JWT Protected Routes
**As a** logged-in user,  
**I want** access to protected bookmark-related routes  
**So that** only authenticated users can view, create, and manage bookmarks.

**Acceptance Criteria:**
- All bookmark CRUD operations require a valid JWT in the request.
- Unauthenticated requests to protected routes receive a 401 Unauthorized response.

---

### User Story 3: Logout Functionality
**As a** user,  
**I want** to log out of my session  
**So that** my JWT token becomes invalid and my session is secure.

**Acceptance Criteria:**
- Users can initiate logout, which removes their token on the client side.
- After logout, accessing protected routes fails (401 Unauthorized).

---

## 3. Scope

**In Scope:**
- User registration and login endpoints
- JWT token issuance upon successful authentication
- Middleware to protect routes using JWT verification
- Logout mechanism (client-side token removal or blacklisting, depending on implementation)
- Updates to API documentation and brief README instructions

**Out of Scope:**
- UI implementation (unless specified otherwise)
- Password reset, email verification, or advanced account management
- Social login (OAuth)

## 4. Estimated Complexity

**Complexity:** Medium

**Reasoning:**  
Implementing secure authentication with JWT involves user data management, careful handling of tokens, and route protection. Established libraries exist, but correct, secure integration requires attention. Logout is straightforward if client-side only but can be more involved if token blacklisting is desired.

## 🏗️ Architecture Design

# Technical Architecture: JWT Authentication for Bookmark Manager

## 1. Overview

The authentication system will be built as a Next.js application deployed on Cloudflare Workers, using the **Edge Runtime** throughout. JWTs will be signed with the **Web Crypto API** (HMAC-SHA256) for Workers compatibility, stored as `HttpOnly` cookies to prevent XSS, and validated via Next.js Middleware before any protected route is reached. User credentials will be persisted in **Cloudflare D1** (SQLite) with passwords hashed via `bcryptjs`.

### Step 3 — JWT Utility (Web Crypto API)

```typescript
// src/lib/auth/jwt.ts
import type { JwtPayload } from '@/types/auth';

const ALGORITHM = { name: 'HMAC', hash: 'SHA-256' };

async function importKey(secret: string): Promise<CryptoKey> {
  const encoded = new TextEncoder().encode(secret);
  return crypto.subtle.importKey('raw', encoded, ALGORITHM, false, ['sign', 'verify']);
}

function base64url(input: ArrayBuffer | string): string {
  const bytes = typeof input === 'string'
    ? new TextEncoder().encode(input)
    : new Uint8Array(input);
  return btoa(String.fromCharCode(...bytes))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}

export async function signJwt(payload: Omit<JwtPayload, 'iat' | 'exp'>, secret: string, expiresInSeconds = 604800): Promise<string> {
  const header = base64url(JSON.stringify({ alg: 'HS256', typ: 'JWT' }));
  const now = Math.floor(Date.now() / 1000);
  const claims: JwtPayload = { ...payload, iat: now, exp: now + expiresInSeconds };
  const body = base64url(JSON.stringify(claims));
  const key = await importKey(secret);
  const signature = await crypto.subtle.sign(
    ALGORITHM.name,
    key,
    new TextEncoder().encode(`${header}.${body}`)
  );
  return `${header}.${body}.${base64url(signature)}`;
}

export async function verifyJwt(token: string, secret: string): Promise<JwtPayload> {
  const parts = token.split('.');
  if (parts.length !== 3) throw new Error('Invalid token structure');
  const [header, body, sig] = parts;
  const key = await importKey(secret);
  const valid = await crypto.subtle.verify(
    ALGORITHM.name,
    key,
    Uint8Array.from(atob(sig.replace(/-/g, '+').replace(/_/g, '/')), c => c.charCodeAt(0)),
    new TextEncoder().encode(`${header}.${body}`)
  );
  if (!valid) throw new Error('Invalid signature');
  const payload: JwtPayload = JSON.parse(atob(body.replace(/-/g, '+').replace(/_/g, '/')));
  if (payload.exp < Math.floor(Date.now() / 1000)) throw new Error('Token expired');
  return payload;
}
```

### Step 4 — Middleware (Route Protection)

```typescript
// src/middleware.ts
import { NextRequest, NextResponse } from 'next/server';
import { verifyJwt } from '@/lib/auth/jwt';

const PROTECTED_PATTERNS = ['/api/bookmarks', '/api/auth/me', '/api/auth/logout'];

export async function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;
  const isProtected = PROTECTED_PATTERNS.some(p => pathname.startsWith(p));
  if (!isProtected) return NextResponse.next();

  const token = request.cookies.get('auth_token')?.value;
  if (!token) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  try {
    const secret = (process.env.JWT_SECRET ?? (request as any).env?.JWT_SECRET)!;
    const payload = await verifyJwt(token, secret);
    const requestHeaders = new Headers(request.headers);
    requestHeaders.set('x-user-id', payload.sub);
    requestHeaders.set('x-user-email', payload.email);
    return NextResponse.next({ request: { headers: requestHeaders } });
  } catch {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
}

export const config = {
  matcher: ['/api/bookmarks/:path*', '/api/auth/me', '/api/auth/logout'],
};
```

### Step 6 — Auth Route Handlers

```typescript
// src/app/api/auth/register/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { hashPassword } from '@/lib/auth/password';
import { signJwt } from '@/lib/auth/jwt';
import { setAuthCookie } from '@/lib/auth/cookies';
import { createUser, findUserByEmail } from '@/lib/db/user.repository';
import { registerSchema } from '@/lib/validation/auth.schemas';
import type { Env } from '@/types/env';

export const runtime = 'edge';

export async function POST(request: NextRequest) {
  const env = (process.env as unknown) as Env;
  const body = await request.json();
  const parsed = registerSchema.safeParse(body);

  if (!parsed.success) {
    return NextResponse.json({ error: 'Validation failed', details: parsed.error.issues }, { status: 400 });
  }

  const { email, password } = parsed.data;
  const existing = await findUserByEmail(env, email);
  if (existing) {
    return NextResponse.json({ error: 'Email already in use' }, { status: 409 });
  }

  const hashed = await hashPassword(password);
  const user = await createUser(env, email, hashed);
  const token = await signJwt({ sub: user.id, email: user.email }, env.JWT_SECRET);

  const response = NextResponse.json(
    { user: { id: user.id, email: user.email, created_at: user.created_at }, message: 'Account created' },
    { status: 201 }
  );
  setAuthCookie(response, token);
  return response;
}
```

## Key Security Decisions

| Decision | Rationale |
|---|---|
| **HttpOnly cookies** over `localStorage` | Prevents XSS-based token theft |
| **HMAC-SHA256 via Web Crypto** | Native edge runtime; no Node.js `crypto` dependency |
| **Generic login error message** | Prevents user enumeration attacks |
| **Ownership check in bookmark routes** | Users can only access their own data even with a valid JWT |
| **`x-user-id` header from middleware** | Single source of truth for identity; route handlers never trust request body for user ID |
| **bcryptjs cost factor 12** | Balances security vs edge CPU limits |

## 4. Concerns

### 🔴 Critical

---

#### C1 — bcryptjs is Not Edge Runtime Compatible
**Category:** Runtime Compatibility / Security  
**Severity:** Critical

`bcryptjs` uses synchronous, CPU-intensive operations. Cloudflare Workers enforce a **CPU time limit of 10ms** for free plans and 30ms for paid plans per request. A bcrypt cost factor of 12 requires approximately **300–400ms of CPU time** on typical hardware. This will cause the registration and login routes to **exceed the CPU limit and fail in production**, not just be slow.

**Evidence from the plan:**
```bash
npm install bcryptjs  # This library is synchronous and CPU-bound
```
```sql
-- cost factor 12  <-- this is the problem
password  TEXT NOT NULL,  -- bcrypt hash, cost factor 12
```

**Recommendation:**  
Replace bcrypt with **Argon2 via WebAssembly** (e.g., `argon2-cloudflare` or `hash-wasm`) designed for the edge, **or** offload password hashing to a Durable Object or external service. If bcryptjs must be used for compatibility reasons, drop the cost factor to 6–8 and document the tradeoff explicitly — but this is not recommended for a new project. Alternatively, use `scrypt` via Web Crypto if it becomes available, or consider **delegating auth entirely** to Cloudflare Access or an external provider (Auth0, Clerk) to sidestep this entirely.

> **Note:** The plan's own Key Security Decisions table acknowledges "Balances security vs edge CPU limits" but does not resolve the conflict — it just names it.

---

#### C2 — JWT Secret Access Pattern is Unreliable
**Category:** Security / Correctness  
**Severity:** Critical

The middleware accesses the JWT secret via:
```typescript
const secret = (process.env.JWT_SECRET ?? (request as any).env?.JWT_SECRET)!;
```

This has two problems:

1. **`process.env` is not available in Cloudflare Workers edge runtime** the same way it is in Node.js. Wrangler injects secrets into the Worker's `env` object, which is passed as a parameter to the `fetch` handler — it is not automatically available on `process.env` in Next.js edge middleware running on Workers.
2. **The `(request as any).env` cast is not a defined API.** This is speculative — `NextRequest` does not expose the Cloudflare `env` binding, and casting with `any` hides a runtime error waiting to happen.

The result: `secret` may be `undefined` at runtime, causing every JWT verification to fail (or worse, to succeed against an empty string if the error is swallowed).

**Recommendation:**  
Use the `@cloudflare/next-on-pages` adapter's documented pattern for accessing env bindings in Next.js edge middleware on Workers, specifically `getRequestContext()`:
```typescript
import { getRequestContext } from '@cloudflare/next-on-pages';

export async function middleware(request: NextRequest) {
  const { env } = getRequestContext();
  const secret = env.JWT_SECRET;
  // ...
}
```
Apply the same pattern consistently across all route handlers. Validate at startup (or in a health-check route) that the secret is present and of acceptable length.

### 🟠 High

#### H1 — No Token Refresh Mechanism
**Category:** Security / UX  
**Severity:** High

The plan issues a 7-day JWT with no refresh strategy. This creates a binary outcome: either the session works fully for 7 days regardless of activity, or it expires and the user is logged out abruptly. There is no mechanism to:
- Extend a session for an active user
- Shorten a session that should be invalidated (compromised credential, password change)
- Detect token reuse after logout (client-side removal only)

**Recommendation:**  
Implement a short-lived access token (15–60 minutes) paired with a longer-lived refresh token (7–30 days) stored in a separate `HttpOnly` cookie. The refresh token should be stored server-side (D1) and invalidated on logout. This is more complex but is the standard approach for session management with JWTs. If this scope is deferred, document it explicitly as a known limitation and reduce the access token TTL to 1 hour rather than 7 days.

---

#### H2 — Logout Does Not Invalidate the Token
**Category:** Security  
**Severity:** High

The product specification states:

> "After logout, accessing protected routes fails (401 Unauthorized)"

The implementation clears the cookie, which satisfies this from the **browser's perspective**. However, the JWT itself remains valid until expiration. If the token was captured (logs, network interception, another device, shared computer), it can be used to make authenticated API requests for up to 7 days after logout.

The spec mentions "blacklisting, depending on implementation" as in-scope. The architecture silently chose client-side only without acknowledging the gap against the stated acceptance criterion.

**Recommendation:**  
For a proper logout story, implement a token denylist in D1 or Cloudflare KV (KV is better suited for this — fast reads, TTL support). Store the `jti` (JWT ID) claim on issuance, add it to the denylist on logout with a TTL matching the token's remaining lifetime. Check the denylist in middleware. This adds one KV read per authenticated request, which is acceptable.

Minimally, add a `jti` claim now (a UUID on each token) so the infrastructure exists to support blacklisting later without reissuing all tokens.

#### H3 — Missing Rate Limiting on Auth Endpoints
**Category:** Security  
**Severity:** High

There is no rate limiting on `/api/auth/register` or `/api/auth/login`. Without it:
- Login is vulnerable to credential stuffing and brute force
- Registration can be abused for spam account creation and D1 storage exhaustion

Cloudflare Workers has native rate limiting capabilities that are not referenced anywhere in the plan.

**Recommendation:**  
Use **Cloudflare Rate Limiting** (configurable in `wrangler.toml` or the dashboard) to apply request limits to the auth routes. A reasonable starting point: 10 requests per minute per IP on login, 5 per minute on register. Additionally, consider implementing exponential backoff responses (429 with `Retry-After` header) for repeated failures from the same IP.

#### H4 — `base64url` Encoding Implementation Has an Edge Case Bug
**Category:** Correctness / Security  
**Severity:** High

The custom `base64url` function in `jwt.ts`:
```typescript
function base64url(input: ArrayBuffer | string): string {
  const bytes = typeof input === 'string'
    ? new TextEncoder().encode(input)
    : new Uint8Array(input);
  return btoa(String.fromCharCode(...bytes))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}
```

The spread `String.fromCharCode(...bytes)` will throw a **`RangeError: Maximum call stack size exceeded`** for large inputs because `Function.prototype.apply` (which spread uses internally) has a call stack argument limit. For JWT headers and small payloads this will rarely trigger, but it is a latent defect.

Additionally, the decode path in `verifyJwt` uses `atob(sig.replace(...))` directly without a corresponding decode helper — the decode logic is duplicated and inconsistent with the encode path.

**Recommendation:**  
Replace `String.fromCharCode(...bytes)` with a safe iterative approach:
```typescript
function uint8ArrayToBase64(bytes: Uint8Array): string {
  let binary = '';
  for (let i = 0; i < bytes.length; i++) {
    binary += String.fromCharCode(bytes[i]);
  }
  return btoa(binary);
}
```
Consider using a well-tested JWT library compatible with the edge runtime (e.g., `jose`) rather than a hand-rolled implementation. The `jose` library explicitly supports Web Crypto and is maintained against the JWT/JWS RFCs.

#### M1 — Email is Stored in JWT Payload and Forwarded as a Header
**Category:** Security / Design  
**Severity:** Medium

```typescript
requestHeaders.set('x-user-email', payload.email);
```

Storing email in the JWT and forwarding it as a plain header means:
- If email changes (future feature), all existing tokens carry stale data
- Headers are logged by many platforms — email addresses in logs is a data minimization concern under GDPR/privacy regulations
- The email header is not used for any authorization decision in the described routes — only `user_id` is needed

**Recommendation:**  
Remove `email` from the JWT payload and from the forwarded header. The `sub` (user ID) claim is sufficient for authorization. If the email is needed in a route handler, look it up from D1 using the user ID. This keeps tokens minimal and avoids propagating PII unnecessarily.

#### M2 — No Input Sanitization Beyond Schema Validation
**Category:** Security  
**Severity:** Medium

The Zod schemas validate format but do not limit input length:
```typescript
email: z.string().email('Invalid email format'),
password: z.string().min(8, ...)
```

A missing `max()` constraint on both fields means:
- Submitting a 10MB "email" string will be parsed, validated, and potentially hashed
- For bcrypt specifically, passwords over 72 bytes are silently truncated — a 1000-character password and its first-72-character prefix are treated as identical

**Recommendation:**  
Add maximum length constraints:
```typescript
email: z.string().email().max(254),          // RFC 5321 limit
password: z.string().min(8).max(72),         // bcrypt silent truncation boundary
```

#### M3 — CSRF Protection Not Addressed
**Category:** Security  
**Severity:** Medium

`SameSite=Strict` provides good CSRF protection for same-origin navigation, but the plan does not explicitly address CSRF for cases where the application might be embedded or accessed cross-origin in the future. There is no mention of CSRF tokens or the `Origin` header validation.

**Recommendation:**  
Document that `SameSite=Strict` is the current CSRF mitigation and its limitations (it does not protect against all CORS-based attack vectors if CORS is ever relaxed). Add an `Origin` header check in the middleware for state-mutating routes as a defense-in-depth measure. If the API is ever consumed by a mobile app or third-party client, this design will need revisitation.

#### M4 — Migration Strategy is Manual and Irreversible
**Category:** Operational  
**Severity:** Medium

The migrations are plain SQL files executed via `wrangler d1 execute`. There is:
- No migration runner tracking which migrations have been applied
- No rollback scripts
- No CI check that migrations are up to date before deployment
- The GitHub Actions workflow modification is mentioned but not specified

**Recommendation:**

Consider using `drizzle-orm` with D1 support, which provides migration tracking, or write a simple migration runner script. Add a CI step that validates the current D1 schema matches the expected state before deploying application code.

#### M5 — `findUserByEmail` Uses `SELECT *`
**Category:** Security / Performance  
**Severity:** Medium

```typescript
return env.DB.prepare(`SELECT * FROM users WHERE email = ?`).bind(email).first<User>();
```

`SELECT *` returns the `password` (bcrypt hash) column in every user lookup, including cases where it is not needed (e.g., `findUserById` for the `/me` endpoint).

**Recommendation:**  
Use explicit column selection. For login, select all columns. For profile lookups, select only `id`, `email`, `created_at`, `updated_at` and return a `PublicUser` type directly from the repository:
```typescript
export async function findUserById(env: Env, id: string): Promise<PublicUser | null> {
  return env.DB.prepare(
    `SELECT id, email, created_at, updated_at FROM users WHERE id = ?`
  ).bind(id).first<PublicUser>();
}
```
This eliminates the risk of accidentally serializing the password hash in a response.

const id = crypto.randomUUID(); // no import needed

#### L2 — Timestamps Use Milliseconds but Convention Comment Says Seconds
**Category:** Correctness  
**Severity:** Low

The schema comment says `-- Unix timestamp (ms)` and the code uses `Date.now()` (milliseconds). The `JwtPayload` uses `iat` and `exp` in Unix **seconds** (as per JWT spec). The inconsistency between DB timestamps (ms) and JWT timestamps (s) is not a bug as-is, but it creates confusion for anyone querying the database and comparing to token expiry.

**Recommendation:** Standardize on one unit. Seconds is the JWT convention; using `Math.floor(Date.now() / 1000)` for DB timestamps as well makes the system consistent and easier to reason about. Update the schema comment accordingly.
