## 📋 Product Specification

# Product Specification: Bookmark Manager API

---

## 1. Summary

The Bookmark Manager API is a secure, user-centric RESTful service enabling users to register, login, and manage their personal bookmarks and tags. Each authenticated user can create, edit, delete, and search bookmarks, organize them via tags, and enjoy private, scalable data storage. The system leverages Node.js, Express, and PostgreSQL, and is designed for containerized deployment.

---

## 2. User Stories

### Story 1: User Registration and Authentication
**As a** new user,  
**I want to** create an account and authenticate with my credentials,  
**So that** my bookmarks are only accessible to me.

**Acceptance Criteria:**
- A user can register with a unique username/email and hashed password.
- A user can login and receive a JWT token for subsequent authenticated requests.
- All bookmark/tag routes enforce JWT authentication and reject unauthenticated requests.

---

### Story 2: Bookmark CRUD and Tagging
**As an** authenticated user,  
**I want to** add, edit, delete, and tag bookmarks,  
**So that** I can organize and manage my saved links efficiently.

**Acceptance Criteria:**
- User can create new bookmarks (URL, title, description, tags).
- User can view, update, and delete only their own bookmarks.
- User can create, assign, remove, and list their own tags.

---

### Story 3: Search Bookmarks
**As an** authenticated user,  
**I want to** search my bookmarks by keyword or tag(s),  
**So that** I can quickly find the information I need.

**Acceptance Criteria:**
- User can query `/bookmarks` with search or tag parameters.
- Only bookmarks belonging to the user are included in results.
- Search supports partial matches on title and description fields, and filtering by tag.

## 🏗️ Architecture Design

# Technical Architecture: Bookmark Manager API

## 1. Overview

A RESTful Node.js/Express API backed by PostgreSQL that enables authenticated users to privately manage bookmarks with tagging and full-text search. JWT middleware protects all resource routes, with user-scoped data isolation enforced at the database query level. The application is fully containerized via Docker Compose, bundling the API server and PostgreSQL instance for portable deployment.

## 3. Data Models

### `users`
```sql
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  username    VARCHAR(50)  UNIQUE NOT NULL,
  email       VARCHAR(255) UNIQUE NOT NULL,
  password_hash TEXT       NOT NULL,
  created_at  TIMESTAMPTZ  DEFAULT NOW()
);
```

### `bookmarks`
```sql
CREATE TABLE bookmarks (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  url         TEXT         NOT NULL,
  title       VARCHAR(255) NOT NULL,
  description TEXT,
  created_at  TIMESTAMPTZ  DEFAULT NOW(),
  updated_at  TIMESTAMPTZ  DEFAULT NOW()
);

-- Indexes for fast user-scoped lookup and full-text search
CREATE INDEX idx_bookmarks_user_id ON bookmarks(user_id);
CREATE INDEX idx_bookmarks_fts ON bookmarks
  USING GIN(to_tsvector('english', title || ' ' || COALESCE(description, '')));
```

### `tags`
```sql
CREATE TABLE tags (
  id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name    VARCHAR(100) NOT NULL,
  UNIQUE(user_id, name)   -- no duplicate tag names per user
);

CREATE INDEX idx_tags_user_id ON tags(user_id);
```

### `bookmark_tags` *(join table)*
```sql
CREATE TABLE bookmark_tags (
  bookmark_id UUID NOT NULL REFERENCES bookmarks(id) ON DELETE CASCADE,
  tag_id      UUID NOT NULL REFERENCES tags(id)      ON DELETE CASCADE,
  PRIMARY KEY (bookmark_id, tag_id)
);

CREATE INDEX idx_bookmark_tags_tag_id ON bookmark_tags(tag_id);
```

## 4. API Endpoints

### Bookmark Routes (`/api/bookmarks`)

| Method | Path | Auth | Description | Query Params | Request Body | Response |
|---|---|---|---|---|---|---|
| `GET` | `/api/bookmarks` | ✅ | List user's bookmarks | `?search=&tag_id=&tag_name=&page=&limit=` | — | `200 { data: Bookmark[], total }` |
| `POST` | `/api/bookmarks` | ✅ | Create new bookmark | — | `CreateBookmarkDto` | `201 Bookmark` |
| `GET` | `/api/bookmarks/:id` | ✅ | Get single bookmark | — | — | `200 Bookmark` |
| `PUT` | `/api/bookmarks/:id` | ✅ | Update bookmark | — | `UpdateBookmarkDto` | `200 Bookmark` |
| `DELETE` | `/api/bookmarks/:id` | ✅ | Delete bookmark | — | — | `204 No Content` |

**Search behavior for `GET /api/bookmarks`:**
- `?search=keyword` → `tsvector` full-text match on `title` + `description`
- `?tag_id=uuid` → filter by specific tag UUID
- `?tag_name=work` → filter by tag name (convenience alias)
- Both params together → AND condition (tag filter + keyword search)

### Tag Routes (`/api/tags`)

| Method | Path | Auth | Description | Request Body | Response |
|---|---|---|---|---|---|
| `GET` | `/api/tags` | ✅ | List all user's tags | — | `200 Tag[]` |
| `POST` | `/api/tags` | ✅ | Create a new tag | `{ name: string }` | `201 Tag` |
| `DELETE` | `/api/tags/:id` | ✅ | Delete a tag (cascades from join table) | — | `204 No Content` |
| `POST` | `/api/bookmarks/:id/tags` | ✅ | Assign tag(s) to bookmark | `{ tag_ids: string[] }` | `200 Bookmark` |
| `DELETE` | `/api/bookmarks/:id/tags/:tagId` | ✅ | Remove tag from bookmark | — | `204 No Content` |

import fs from 'fs';
import path from 'path';
import { pool } from '../../config/db';

export async function runMigrations() {
  await pool.query(`
    CREATE TABLE IF NOT EXISTS _migrations (name TEXT PRIMARY KEY, run_at TIMESTAMPTZ DEFAULT NOW())
  `);
  const dir = path.join(__dirname);
  const files = fs.readdirSync(dir)
    .filter(f => f.endsWith('.sql'))
    .sort();

  for (const file of files) {
    const { rows } = await pool.query('SELECT 1 FROM _migrations WHERE name = $1', [file]);
    if (rows.length === 0) {
      const sql = fs.readFileSync(path.join(dir, file), 'utf8');
      await pool.query(sql);
      await pool.query('INSERT INTO _migrations(name) VALUES($1)', [file]);
      console.log(`[migrate] ran: ${file}`);
    }
  }
}

### Key Implementation Notes

> **Ownership enforcement:** Every repository query for read/update/delete includes `AND user_id = $userId`. Never rely solely on the route parameter — always scope to the authenticated user.

> **Tag sync on bookmark update:** Use a PostgreSQL transaction — DELETE existing `bookmark_tags` rows for the bookmark, then INSERT the new set. This avoids partial state on failure.

> **Search query:** Use `plainto_tsquery` (not `to_tsquery`) — it safely handles arbitrary user input without special operator syntax.

> **Pagination:** Default `limit=20, page=1`. Return `{ data, total, page, limit }` shape so clients can implement pagination UI.

## 4. Concerns

### 🔴 CRITICAL — No JWT Revocation / Logout Mechanism

**Category:** Authentication  
**Description:** The spec mentions "JWT login/logout" in scope, but no logout endpoint exists in the API table, and there is no token blacklist, revocation list, or short-expiry + refresh token pattern described anywhere. A stolen JWT remains valid until expiry with no recourse.  
**Recommendation:** Either (a) implement a token blacklist table with a `POST /api/auth/logout` endpoint that stores the JTI (JWT ID) until expiry — check the blacklist in `requireAuth` middleware — or (b) use short-lived access tokens (15 minutes) plus a refresh token flow with a `refresh_tokens` table. Option (a) is simpler for this scope. Add `jti` claim to JWT payload with `crypto.randomUUID()`.

```typescript
// JWT payload shape should be explicit
interface JwtPayload {
  sub: string;   // userId
  jti: string;   // unique token ID for revocation
  iat: number;
  exp: number;
}
```

---

### 🔴 CRITICAL — No Rate Limiting on Auth Endpoints

**Category:** Security / Abuse Prevention  
**Description:** The spec explicitly lists rate limiting as "out of scope," but the auth endpoints (`/register`, `/login`) are brute-force targets. Bcrypt intentionally slows password comparison — 12 rounds at ~250ms per attempt means an attacker can still try ~4 passwords/second per connection. No limit means credential stuffing is trivially automated.  
**Recommendation:** Add `express-rate-limit` specifically to auth routes only — this is a single middleware, not "advanced security." It does not need to be excluded from scope.

```typescript
import rateLimit from 'express-rate-limit';

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 20,
  standardHeaders: true,
  legacyHeaders: false,
  message: { error: 'Too many requests, try again later' }
});

app.use('/api/auth', authLimiter, authRouter);
```

---

### 🔴 CRITICAL — JWT Secret in Docker Compose Environment

**Category:** Secrets Management  
**Description:** `docker-compose.yml` shows `JWT_SECRET: ${JWT_SECRET}` which relies on the shell environment having this value set. If `.env` is used at compose level without explicit exclusion, or if someone runs `docker compose up` without the var set, the behavior is undefined (or it fails open to an empty string — Zod's `min(32)` check saves this at startup, but only if the validation runs correctly).  
**Recommendation:** Document explicitly that `.env` must never be committed. Add `.env` to `.gitignore` and verify it's there. For production, use Docker secrets or a secrets manager (AWS SSM, HashiCorp Vault). The `.env.example` should contain an obviously invalid placeholder: `JWT_SECRET=REPLACE_ME_WITH_64_CHAR_RANDOM_HEX`.

---

### 🟠 HIGH — Migration Runner Has No Transaction Wrapping

**Category:** Data Integrity  
**Description:** The `runMigrations()` function runs each SQL file and then inserts into `_migrations` as two separate queries. If the process crashes between these two lines, the migration is applied but not recorded — running migrations again will attempt to re-apply it and likely fail or corrupt data.  
**Recommendation:** Wrap each migration in a transaction:

```typescript
const client = await pool.connect();
try {
  await client.query('BEGIN');
  await client.query(sql);
  await client.query('INSERT INTO _migrations(name) VALUES($1)', [file]);
  await client.query('COMMIT');
} catch (err) {
  await client.query('ROLLBACK');
  throw err;
} finally {
  client.release();
}
```

**Also:** Consider using an established migration library (`node-pg-migrate`, `db-migrate`) rather than a custom runner. The edge cases in custom migration runners are well-documented and painful.

---

### 🟠 HIGH — URL Field Has No Validation

**Category:** Input Validation / Data Integrity  
**Description:** The `url` field in `CreateBookmarkDto` is typed as `string` but there is no indication that Zod will validate it as an actual URL. A user could store arbitrary strings, JavaScript URIs (`javascript:alert(1)`), or malformed values.  
**Recommendation:** In `bookmarks.schema.ts`, enforce URL validation:

```typescript
const CreateBookmarkSchema = z.object({
  url:         z.string().url().max(2048),
  title:       z.string().min(1).max(255).trim(),
  description: z.string().max(5000).trim().optional(),
  tag_ids:     z.array(z.string().uuid()).max(50).optional(),
});
```

`z.string().url()` rejects `javascript:` URIs. Also add a `max(2048)` — unbounded URL fields are a minor DoS vector.

---

### 🟠 HIGH — Tag IDs From User Input Are Not Ownership-Validated

**Category:** Authorization / Insecure Direct Object Reference  
**Description:** `CreateBookmarkDto` accepts `tag_ids: string[]`. If a user submits a `tag_id` that belongs to a different user, the application will either throw a foreign key constraint error (leaking that the ID exists) or silently associate another user's tag with their bookmark.  
**Recommendation:** Before inserting into `bookmark_tags`, validate that all provided `tag_ids` belong to the authenticated user:

```typescript
// In bookmarks.service.ts or repository.ts
const { rows } = await client.query(
  `SELECT id FROM tags WHERE id = ANY($1::uuid[]) AND user_id = $2`,
  [tag_ids, userId]
);
if (rows.length !== tag_ids.length) {
  throw new ForbiddenError('One or more tag IDs are invalid or not owned by user');
}
```

### 🟠 HIGH — Error Middleware Specification Is Incomplete

**Category:** Security / Information Disclosure  
**Description:** `error.middleware.ts` is listed in the file structure but never specified. Without explicit guidance, a developer will likely use a default implementation that leaks stack traces and internal error details in production.  
**Recommendation:** Specify the error handler behavior explicitly:

```typescript
export const errorMiddleware: ErrorRequestHandler = (err, req, res, next) => {
  const isDev = env.NODE_ENV === 'development';
  
  // Log internally always
  console.error(err);
  
  // Known application errors
  if (err instanceof AppError) {
    return res.status(err.statusCode).json({ error: err.message });
  }
  
  // Never leak internals in production
  res.status(500).json({
    error: isDev ? err.message : 'Internal server error',
    ...(isDev && { stack: err.stack })
  });
};
```

Define a typed `AppError` base class with `statusCode` and `message` that services throw.

### 🟡 MEDIUM — No JWT Expiry Specified

**Category:** Authentication  
**Description:** The plan never states what the JWT expiry (`exp`) should be. A developer left to decide will likely pick a large value (7 days, 30 days) "to avoid user friction."  
**Recommendation:** Specify explicitly in the plan: `expiresIn: '24h'` for a reasonable single-token flow, or `'15m'` if implementing refresh tokens. Document this in `jwt.ts`:

```typescript
export const signToken = (userId: string, jti: string): string =>
  jwt.sign({ sub: userId, jti }, env.JWT_SECRET, { expiresIn: '24h', algorithm: 'HS256' });
```

Explicitly pin the algorithm to prevent the `alg: none` attack class.

### 🟡 MEDIUM — `tag_name` Search Parameter Creates Ambiguity

**Category:** API Design  
**Description:** `GET /api/bookmarks?tag_name=work` as a "convenience alias" for `tag_id` introduces ambiguity. Tag names are not globally unique — only `UNIQUE(user_id, name)`. While user scoping prevents cross-user collision, the behavior when zero or multiple matches exist is undefined. Also, the query complexity increases (join to resolve name → ID).  
**Recommendation:** Remove `tag_name` from v1. Keep only `tag_id`. If tag name search is desired, add a separate `GET /api/tags?name=work` to resolve IDs first. Alternatively, document the exact behavior: return empty result if tag name doesn't match any of the user's tags; never error.

### 🟡 MEDIUM — Pagination Is Unbounded Without Explicit MAX

**Category:** Performance / DoS  
**Description:** `Default limit=20` is good, but there is no specified maximum. A client sending `?limit=100000` will force PostgreSQL to scan the entire table and return a massive payload.  
**Recommendation:** In `pagination.ts`, cap the limit:

```typescript
export const parsePagination = (query: Record<string, unknown>) => {
  const page  = Math.max(1, parseInt(String(query.page  ?? 1)));
  const limit = Math.min(100, Math.max(1, parseInt(String(query.limit ?? 20))));
  const offset = (page - 1) * limit;
  return { page, limit, offset };
};
```

### 🟡 MEDIUM — `ON CONFLICT DO NOTHING` on Tag Creation Silently Swallows Data

**Category:** API Design / Developer Experience  
**Description:** `INSERT ... ON CONFLICT DO NOTHING` returns no rows on conflict. The calling code will receive an empty result and likely throw a null reference error or return a confusing empty 201 response.  
**Recommendation:** Use `ON CONFLICT DO UPDATE` to return the existing row:

```typescript
// tags.repository.ts
INSERT INTO tags (user_id, name) VALUES ($1, $2)
ON CONFLICT (user_id, name) DO UPDATE SET name = EXCLUDED.name
RETURNING *
```

This is idempotent and always returns the tag, whether newly created or pre-existing. Return `200` if pre-existing, `201` if newly created (check `rowCount` before conflict resolution, or use a CTE).

### 🟡 MEDIUM — No HTTPS / TLS Configuration Mentioned

**Category:** Transport Security  
**Description:** The Dockerfile exposes port 3000 with no TLS. For production deployment there is no mention of TLS termination.  
**Recommendation:** Document explicitly that TLS termination is expected at the reverse proxy layer (nginx, Caddy, or cloud load balancer). Add `TRUST_PROXY=1` note for `express` when behind a proxy, and call `app.set('trust proxy', 1)` — this matters for rate limiting (getting real IP) and secure cookies.

### 🟢 LOW — `POSTGRES_USER` Not Set in Docker Compose

**Category:** Configuration  
**Description:** `docker-compose.yml` sets `POSTGRES_PASSWORD` and `POSTGRES_DB` but not `POSTGRES_USER`. PostgreSQL defaults to `postgres` as the superuser, which the `DATABASE_URL` also assumes. This works but is implicit.  
**Recommendation:** Make it explicit: `POSTGRES_USER: postgres`. For production, create a least-privilege application user with only `SELECT, INSERT, UPDATE, DELETE` on application tables — not the PostgreSQL superuser.

### 🟢 LOW — No `updated_at` Trigger on Bookmarks

**Category:** Data Integrity  
**Description:** The `bookmarks` table has `updated_at TIMESTAMPTZ DEFAULT NOW()` but no trigger or application-level update. A `PUT` on a bookmark will not update `updated_at` unless explicitly included in the UPDATE statement. Developers may forget.  
**Recommendation:** Add a PostgreSQL trigger, or document that the UPDATE query must always include `updated_at = NOW()`. A trigger is more reliable:

```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER bookmarks_updated_at
BEFORE UPDATE ON bookmarks
FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

### 🟢 LOW — Test Isolation Strategy Unspecified

**Category:** Testing  
**Description:** Integration tests against a shared PostgreSQL instance will interfere with each other unless isolation is handled. The plan doesn't specify how test state is managed between tests.  
**Recommendation:** Use one of: (a) `TRUNCATE` all tables in a `beforeEach` block, (b) wrap each test in a transaction that is rolled back, or (c) use separate schemas per test suite. Option (a) is simplest:

```typescript
beforeEach(async () => {
  await pool.query('TRUNCATE users, bookmarks, tags, bookmark_tags CASCADE');
});
```
