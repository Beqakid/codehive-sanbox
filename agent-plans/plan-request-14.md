# Product Specification: Expense Tracker API

## 1. Summary

The Expense Tracker API is a RESTful backend service enabling a single authenticated user to manage personal expenses, categorize transactions, and retrieve monthly spending summaries. The system’s core features include secure login via JWT, CRUD operations on expenses, and monthly totals categorized for easy budgeting. Data is persistently stored in local JSON files, leveraging TypeScript and Express.js for robust, type-safe development.

**As** a user,  
**I want** to add, view, edit, and delete expenses with categories,  
**So that** I can keep track of where my money goes.

**As** a user,  
**I want** to log in with my credentials and receive a JWT,  
**So that** I can securely access and manage my expenses.  
*Acceptance Criteria:*
- Login accepts a username and password and returns a JWT if credentials are correct.
- All other endpoints require a valid JWT for access.

### Story 3: Monthly Summary  
**As** a user,  
**I want** to retrieve a monthly summary of expenses broken down by category,  
**So that** I can see where I spend the most each month.  
*Acceptance Criteria:*

- Only authenticated requests are processed.

# Technical Architecture: Expense Tracker API

## 1. Overview

A single-user REST API built with Express.js and TypeScript that handles expense CRUD operations, JWT authentication, and monthly spending summaries. All data is persisted to local JSON files (`expenses.json`, `user.json`) with a simple file-locking utility to prevent write corruption. The application is structured with clear separation between routes, middleware, models, and storage utilities.

## 4. Concerns

### 🔴 HIGH — File Lock Race Condition Gap
**Category:** Correctness / Data Integrity

`proper-lockfile` locks on *write*, but read-modify-write sequences (PUT, DELETE, POST) require the lock to span the *entire read → mutate → write cycle*, not just the write. As currently described:

```
read()     ← no lock held
mutate()   ← no lock held  
writeJSON() ← lock acquired here, too late
```

Two concurrent requests (e.g., two rapid DELETEs) can both read the same state before either writes, causing the second write to resurrect a deleted record.

**Recommendation:** Acquire the lock at the start of `writeJSON` and expose a `withLock(filePath, async () => { read → mutate → write })` wrapper. Route handlers should use this transaction helper for all mutating operations.

```typescript
// utils/fileStorage.ts — proposed wrapper
export async function withFileLock<T>(
  filePath: string,
  fn: () => Promise<T>
): Promise<T> {
  const release = await lockfile.lock(filePath, { retries: 3 });
  try {
    return await fn();
  } finally {
    await release();
  }
}
```

### 🔴 HIGH — JWT Secret Startup Validation Missing
**Category:** Security / Operational Reliability

If `JWT_SECRET` is missing or empty in `.env`, `jwt.sign` and `jwt.verify` will either throw unpredictably at runtime or, worse, silently use an empty string as the secret — making every forged token valid.

**Recommendation:** Add a startup guard in `server.ts`:

```typescript
const secret = process.env.JWT_SECRET;
if (!secret || secret.length < 32) {
  console.error('FATAL: JWT_SECRET is missing or too short. Exiting.');
  process.exit(1);
}
```

### 🟠 MEDIUM — `expenses.json` Excluded from `.gitignore` Would Be a Data Leak; Included Would Break Setup
**Category:** Developer Experience / Security

The plan states `.gitignore` excludes `data/` entirely. This creates two conflicting problems:

1. `data/expenses.json` (initialized as `[]`) and `data/user.json` need to *exist* in the repo as committed empty/placeholder files so the app doesn't crash on first run.
2. But `data/user.json` with real credentials must *not* be committed.

**Recommendation:** Commit `data/expenses.json` (empty array) and a `data/user.json.example` as a template. Add only `data/user.json` to `.gitignore` — not the entire `data/` directory. Alternatively, have `fileStorage.ts` auto-initialize missing files on first read.

---

### 🟠 MEDIUM — Summary Endpoint Date Filtering is Fragile
**Category:** Correctness

```
Filter expenses where date starts with "YYYY-MM"
```

This string-prefix match works if all dates are in `"2025-06-15T00:00:00.000Z"` format, but will silently break for:

- Dates stored in local time (e.g., `"2025-06-30T23:00:00-05:00"` — UTC would be July)
- Any date format deviation introduced by future clients

**Recommendation:** Parse the date field with `new Date()` during the filter and compare `year`/`month` using UTC getters explicitly:

```typescript
const [year, month] = queryMonth.split('-').map(Number);
const match = expenses.filter(e => {
  const d = new Date(e.date);
  return d.getUTCFullYear() === year && d.getUTCMonth() + 1 === month;
});
```

---

### 🟠 MEDIUM — No Rate Limiting on `/login`
**Category:** Security

The `/login` endpoint does bcrypt comparison on every request. Without rate limiting, it is trivially brute-forceable — bcrypt slows individual attempts but doesn't prevent parallelized or sequential attacks from a single IP.

**Recommendation:** Add `express-rate-limit` scoped specifically to `/login`:

```typescript
import rateLimit from 'express-rate-limit';
const loginLimiter = rateLimit({ windowMs: 15 * 60 * 1000, max: 10 });
app.use('/login', loginLimiter);
```

This adds one dependency and three lines. Worth it even for a local tool since localhost-exposed APIs are routinely tested by automated scanners.

---

### 🟠 MEDIUM — `PUT /expenses/:id` Merges Without Re-validating the Final Object
**Category:** Correctness / Data Integrity

The plan says "validate → merge" for PUT, but if partial validation passes individual fields while the merge produces an invalid combination, the stored record could become inconsistent. More critically: if `amount` passes as a valid positive number but is provided as a string by the client (common with form submissions), Zod's `z.number()` will reject it — but the error message may not be clear enough.

**Recommendation:** Validate the *merged result* after applying the patch, not just the incoming partial payload. Add `z.coerce.number().positive()` for amount to handle string-encoded numbers gracefully.

---

### 🟡 LOW — No `PATCH` — Using `PUT` for Partial Updates is Semantically Incorrect
**Category:** API Design

The spec uses `PUT /expenses/:id` with an `UpdateExpenseInput = Partial<CreateExpenseInput>` body (all fields optional). `PUT` semantically means full replacement. Using `PUT` for partial updates that skip unchanged fields is technically wrong and will confuse API consumers or static analysis tools.

**Recommendation:** Rename to `PATCH /expenses/:id`. This is a one-word change in the route definition with no logic impact, but it correctly signals partial update semantics.

---

### 🟡 LOW — `DELETE /expenses/:id` Returns `200` Instead of `204`
**Category:** API Design

Returning `200` with `{ success: true, data: { id } }` is acceptable but non-standard. `DELETE` success is conventionally `204 No Content`.

**Recommendation:** Use `204` with no body, or if the response envelope is important for client consistency, document the `200` choice explicitly. Either is fine; inconsistency with HTTP conventions is the concern.

---

### 🟡 LOW — `uuid` v9 Default Export Changed
**Category:** Implementation Correctness

`uuid` v9 changed the import style. `uuid()` will not work — it must be `import { v4 as uuidv4 } from 'uuid'`. This is a minor but compilation-breaking detail if the implementation team doesn't check the v9 migration guide.

**Recommendation:** Make the import explicit in the implementation step:

```typescript
import { v4 as uuidv4 } from 'uuid';
const id = uuidv4();
```

---

### 🟡 LOW — Seed Script Accepts Password via CLI Argument
**Category:** Security / Operational

`npx ts-node scripts/seedUser.ts admin mypassword` — CLI arguments are logged in shell history (`~/.bash_history`, `~/.zsh_history`) and visible in process lists (`ps aux`).

**Recommendation:** Read the password from `stdin` (prompt) or from an environment variable instead:

```typescript
import * as readline from 'readline';
const rl = readline.createInterface({ input: process.stdin, output: process.stdout });
rl.question('Password: ', async (password) => { /* hash and write */ });
```

## 6. Scalability Notes

> *Note: The spec explicitly marks cloud deployment and scaling as out of scope. These notes are forward-looking flags, not current requirements.*

| Concern | Severity | Note |
|---|---|---|
| **File-based persistence** | Low (current scope) | JSON flat files are the correct choice for a local single-user tool. They become a hard blocker at multi-user or concurrent-session scale. Document the migration path to SQLite (better-sqlite3) clearly. |
