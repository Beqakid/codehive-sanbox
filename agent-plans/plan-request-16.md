# Product Specification: User Authentication for Bookmark Manager

## 1. Summary
We will implement user authentication for the Bookmark Manager using JWT (JSON Web Tokens). This includes user login and logout functionality, and ensures only authenticated users can access protected routes. The goal is to secure bookmarks and user data with role-appropriate access control.

## 2. User Stories

### User Story 1: Login and Logout
**As** a user,  
**I want** to securely log in with my credentials and log out when I'm done,  
**So that** my bookmark data is protected and only accessible to me.

**Acceptance Criteria:**
- User can submit a username/email and password to log in.
- On successful login, the system returns a JWT token.
- User can log out, which invalidates the current session/client token.
- Error message shown on invalid login.

### User Story 2: Protected Routes
**As** a user,  
**I want** my bookmarks to be accessible only when I’m authenticated,  
**So that** unauthorized users cannot view or modify my bookmarks.

**Acceptance Criteria:**
- Bookmark management routes (e.g., view, add, delete) require a valid JWT.
- Requests without a valid token receive an authorization error.
- Token is verified on each protected API request.

## 3. Scope

### In Scope
- JWT-based authentication implementation (issue, verify, invalidate tokens).
- Login and logout API endpoints.
- Middleware to protect bookmark-related routes.
- Documentation update for authentication flows.

### Out of Scope
- User registration and password recovery flows.
- Social login or OAuth integrations.
- UI/UX design for authentication (unless specifically requested).
- Rate limiting or advanced security controls (beyond JWT).

## 4. Estimated Complexity

**Medium**  
*Reasoning*: JWT authentication is a well-established pattern with many library supports, but requires coordination between login, token generation, middleware protection, and error handling for clean user experience and security. Protecting routes and managing sessions adds moderate development and testing effort.

# Technical Architecture: Bookmark Manager — JWT Authentication

## 1. Overview

We will build a JWT-based authentication system on a Next.js application with Cloudflare Workers handling API routes at the edge. Short-lived access tokens (15 min) paired with HTTP-only refresh token cookies provide a secure, stateless session model. A middleware layer intercepts all protected bookmark routes and validates tokens before forwarding requests.

### Step 2 — JWT Service (`lib/auth/jwtService.ts`)
```typescript
import { SignJWT, jwtVerify } from "jose";
import type { JWTPayload } from "@/lib/types/auth";

const ACCESS_SECRET = new TextEncoder().encode(process.env.JWT_SECRET!);
const REFRESH_SECRET = new TextEncoder().encode(process.env.JWT_REFRESH_SECRET!);

export async function signAccessToken(payload: Omit<JWTPayload, "iat" | "exp" | "jti">) {
  return new SignJWT({ ...payload })
    .setProtectedHeader({ alg: "HS256" })
    .setIssuedAt()
    .setExpirationTime("15m")
    .setJti(crypto.randomUUID())
    .sign(ACCESS_SECRET);
}

export async function verifyAccessToken(token: string): Promise<JWTPayload> {
  const { payload } = await jwtVerify(token, ACCESS_SECRET);
  return payload as JWTPayload;
}

export async function signRefreshToken(userId: string, jti: string) {
  return new SignJWT({ sub: userId })
    .setProtectedHeader({ alg: "HS256" })
    .setIssuedAt()
    .setExpirationTime("7d")
    .setJti(jti)
    .sign(REFRESH_SECRET);
}

export async function verifyRefreshToken(token: string) {
  const { payload } = await jwtVerify(token, REFRESH_SECRET);
  return payload;
}
```

### Step 3 — Auth Middleware (`middleware.ts`)
```typescript
import { NextRequest, NextResponse } from "next/server";
import { verifyAccessToken } from "@/lib/auth/jwtService";

const PROTECTED_PATHS = ["/api/bookmarks"];

export async function middleware(req: NextRequest) {
  const isProtected = PROTECTED_PATHS.some((p) => req.nextUrl.pathname.startsWith(p));
  if (!isProtected) return NextResponse.next();

  const authHeader = req.headers.get("Authorization");
  const token = authHeader?.replace("Bearer ", "");

  if (!token) {
    return NextResponse.json({ error: "Unauthorized", code: "NO_TOKEN" }, { status: 401 });
  }

  try {
    const payload = await verifyAccessToken(token);
    // Forward user identity to API route via header
    const requestHeaders = new Headers(req.headers);
    requestHeaders.set("x-user-id", payload.sub);
    requestHeaders.set("x-user-role", payload.role);
    return NextResponse.next({ request: { headers: requestHeaders } });
  } catch {
    return NextResponse.json({ error: "Unauthorized", code: "TOKEN_INVALID" }, { status: 401 });
  }
}

export const config = {
  matcher: ["/api/bookmarks/:path*"],
};
```

### Step 4 — Login API Route (`app/api/auth/login/route.ts`)
```typescript
import { NextRequest, NextResponse } from "next/server";
import { findUserByEmail } from "@/lib/db/userRepository";
import { comparePassword } from "@/lib/auth/passwordService";
import { signAccessToken, signRefreshToken } from "@/lib/auth/jwtService";
import { createRefreshToken } from "@/lib/db/refreshTokenRepository";
import { REFRESH_TOKEN_COOKIE } from "@/lib/auth/constants";

export async function POST(req: NextRequest) {
  const { email, password } = await req.json();

  const user = await findUserByEmail(email);
  if (!user || !(await comparePassword(password, user.password_hash))) {
    return NextResponse.json({ error: "Invalid credentials" }, { status: 401 });
  }

  const jti = crypto.randomUUID();
  const [accessToken, refreshToken] = await Promise.all([
    signAccessToken({ sub: user.id, email: user.email, username: user.username, role: user.role }),
    signRefreshToken(user.id, jti),
  ]);

  await createRefreshToken({ id: jti, userId: user.id, rawToken: refreshToken });

  const response = NextResponse.json({
    accessToken,
    user: { id: user.id, email: user.email, username: user.username, role: user.role },
  });

  response.cookies.set(REFRESH_TOKEN_COOKIE, refreshToken, {
    httpOnly: true,
    secure: true,
    sameSite: "strict",
    path: "/api/auth",
    maxAge: 60 * 60 * 24 * 7, // 7 days
  });

  return response;
}
```

import { createContext, useCallback, useState } from "react";
import type { User } from "@/lib/types/auth";

export function AuthProvider({ children }: { children: React.ReactNode }) {

  const login = useCallback(async (email: string, password: string) => {

      headers: { "Content-Type": "application/json" },

  return <AuthContext.Provider value={{ ...state, login, logout }}>{children}</AuthContext.Provider>;
}

## 3. Strengths

- **Refresh token hashing** — Storing `SHA-256(token)` in D1 rather than the raw token is correct. Prevents database compromise from yielding usable tokens.
- **Dual-secret separation** — Using distinct secrets (`JWT_SECRET` vs `JWT_REFRESH_SECRET`) for access and refresh tokens is a meaningful defense-in-depth measure.
- **Cookie security attributes** — `HttpOnly`, `Secure`, `SameSite=Strict`, scoped `Path=/api/auth`. This is well-configured and limits XSS/CSRF exposure.
- **Access token in memory** — Storing the access token in React state (not `localStorage`) avoids the most common XSS theft vector.
- **Token revocation table** — The `refresh_tokens` table with a `revoked` flag enables server-side invalidation, which pure stateless JWTs cannot provide.
- **Ownership checks are called out** — The testing checklist includes cross-user access (`403`), which is often forgotten at the design stage.
- **`jose` library choice** — Correct choice for edge/Workers runtime. Native crypto; no Node.js dependencies.
- **File structure** — Well-organized, clearly maps spec intent to concrete files. Migrations are versioned.

## 4. Concerns

### 🔴 Critical

---

#### C1 — Refresh Token Rotation Not Implemented
**Category:** Security / Authentication  
**Severity:** Critical

The `/api/auth/refresh` route is listed in the spec but never implemented. More importantly, there is no design for **refresh token rotation** — issuing a new refresh token on every use and invalidating the old one.

Without rotation, a stolen refresh token is valid for the full 7-day window with no detection mechanism.

**Recommendation:**
```typescript
// On /api/auth/refresh:
// 1. Verify the incoming refresh token against DB
// 2. Check revoked = false AND expires_at > now()
// 3. Revoke the OLD token (set revoked = 1)
// 4. Issue a NEW refresh token with a new jti
// 5. Store the new token hash in refresh_tokens
// 6. Set new cookie + return new access token
```

Implement **reuse detection**: if a revoked token is presented, revoke the entire token family for that user (invalidate all their refresh tokens). This detects token theft.

---

#### C2 — `bcrypt` Will Not Run on Cloudflare Workers
**Category:** Runtime Compatibility  
**Severity:** Critical

`bcrypt` requires Node.js native bindings. Cloudflare Workers run on the V8 isolate runtime — **`bcrypt` will throw at runtime**, not at build time. This will silently pass local development (Node.js) and fail in production.

**Recommendation:**

Use `bcryptjs` (pure JS — will work but is slow) or, preferably, the **Web Crypto API** with `PBKDF2` or `scrypt` which are available natively in the Workers runtime:

```typescript
// passwordService.ts — using Web Crypto PBKDF2 (Workers-compatible)
const ITERATIONS = 200_000;
const KEY_LENGTH = 32;
const DIGEST = "SHA-256";

export async function hashPassword(password: string): Promise<string> {
  const salt = crypto.getRandomValues(new Uint8Array(16));
  const keyMaterial = await crypto.subtle.importKey(
    "raw", new TextEncoder().encode(password), "PBKDF2", false, ["deriveBits"]
  );
  const derived = await crypto.subtle.deriveBits(
    { name: "PBKDF2", salt, iterations: ITERATIONS, hash: DIGEST },
    keyMaterial, KEY_LENGTH * 8
  );
  // Store as "salt:hash" both base64-encoded
  return `${btoa(String.fromCharCode(...salt))}:${btoa(String.fromCharCode(...new Uint8Array(derived)))}`;
}
```

Document the chosen algorithm and parameters explicitly so they can be audited.

#### H1 — No Input Validation on Login Endpoint
**Category:** Security / Input Handling  
**Severity:** High

The login handler does `const { email, password } = await req.json()` with no validation. If `req.json()` throws (malformed body), the route crashes with an unhandled 500. If `email` is an object or array (prototype pollution attempt), downstream behavior is undefined.

**Recommendation:**

Add a schema validation layer before any business logic. `zod` is Workers-compatible:

```typescript
import { z } from "zod";

const LoginSchema = z.object({
  email: z.string().email().max(254),
  password: z.string().min(8).max(128),
});

export async function POST(req: NextRequest) {
  let body: unknown;
  try {
    body = await req.json();
  } catch {
    return NextResponse.json({ error: "Invalid request body" }, { status: 400 });
  }

  const result = LoginSchema.safeParse(body);
  if (!result.success) {
    return NextResponse.json({ error: "Invalid input" }, { status: 400 });
  }
  const { email, password } = result.data;
  // ...
}
```

Apply the same pattern to all API routes that accept a body.

#### H2 — `x-user-id` Header Trust Problem
**Category:** Security / Authorization  
**Severity:** High

The middleware forwards identity via `x-user-id` and `x-user-role` headers. API routes that read `req.headers.get("x-user-id")` are implicitly trusting these headers. If any route is ever **directly accessible** (bypassing middleware — e.g., during local dev, misconfigured matcher, or a future route added without updating `PROTECTED_PATHS`), an attacker can spoof identity by setting this header directly.

**Recommendation:**

Two mitigations, both should be applied:

1. **Strip incoming `x-user-*` headers** at the middleware boundary before forwarding — never let client-supplied values of these headers reach API routes:

```typescript
// In middleware.ts
const requestHeaders = new Headers(req.headers);
requestHeaders.delete("x-user-id");   // strip any client-supplied value
requestHeaders.delete("x-user-role");
// ...then set your own after verification
requestHeaders.set("x-user-id", payload.sub);
requestHeaders.set("x-user-role", payload.role);
```

2. **Never rely solely on headers for authorization** in ownership checks. Re-fetch the user from DB or verify ownership via the JWT payload directly for sensitive mutations.

#### H3 — `ALTER TABLE bookmarks` Migration Is Destructive
**Category:** Data Integrity  
**Severity:** High

Migration `002` adds `user_id TEXT NOT NULL` to an existing `bookmarks` table. If any rows exist, this migration will **fail** because existing rows cannot satisfy `NOT NULL`. In SQLite (D1), `ALTER TABLE ... ADD COLUMN` with `NOT NULL` and no `DEFAULT` is rejected outright.

**Recommendation:**

Choose one:

```sql
-- Option A: Provide a sentinel value (for dev/test data)
ALTER TABLE bookmarks ADD COLUMN user_id TEXT NOT NULL DEFAULT 'UNASSIGNED';

-- Option B: Two-step migration (correct for production)
-- Step 1: Add nullable
ALTER TABLE bookmarks ADD COLUMN user_id TEXT;
-- Step 2: Backfill (manually or via script)
-- UPDATE bookmarks SET user_id = '<admin-user-id>' WHERE user_id IS NULL;
-- Step 3: Enforce NOT NULL in a new migration after backfill
-- (SQLite doesn't support ADD CONSTRAINT, so recreate the table)
```

Document the migration strategy explicitly — especially for a deployment that may already have data.

---

#### H4 — No Rate Limiting on Auth Endpoints
**Category:** Security / Brute Force  
**Severity:** High

The spec explicitly places rate limiting "out of scope," but this is a mistake for auth endpoints specifically. `/api/auth/login` without rate limiting is directly exploitable for credential stuffing and brute force. This is not an "advanced" security control — it is a baseline for any public-facing login endpoint.

**Recommendation:**

At minimum, implement Cloudflare Workers-native rate limiting using the **Rate Limiting API** (available in Workers) or Cloudflare's built-in WAF rules at the zone level. A simple per-IP counter in Durable Objects or KV is also viable:

```typescript
// Minimal KV-based rate limit check in login route
const key = `ratelimit:login:${clientIP}`;
const attempts = parseInt(await env.KV.get(key) ?? "0");
if (attempts >= 10) {
  return NextResponse.json({ error: "Too many attempts" }, { status: 429 });
}
await env.KV.put(key, String(attempts + 1), { expirationTtl: 900 }); // 15-min window
```

Remove this from "out of scope" for the auth subsystem specifically.

#### M1 — Logout Does Not Revoke Server-Side Token
**Category:** Security / Session Management  
**Severity:** Medium

The `POST /api/auth/logout` spec says it "clears the refresh token cookie" and "invalidates client-side state." But there is no mention of marking the refresh token as `revoked = 1` in D1. Clearing a cookie client-side does not invalidate it — if an attacker captured it, they can still use it until expiry.

**Recommendation:**

```typescript
// logout/route.ts
export async function POST(req: NextRequest) {
  const refreshToken = req.cookies.get(REFRESH_TOKEN_COOKIE)?.value;
  if (refreshToken) {
    const tokenHash = await sha256(refreshToken);
    await revokeRefreshTokenByHash(tokenHash); // set revoked = 1 in D1
  }
  const response = NextResponse.json({ success: true });
  response.cookies.delete(REFRESH_TOKEN_COOKIE);
  return response;
}
```

#### M2 — `AuthProvider` Does Not Handle Token Expiry / Silent Refresh
**Category:** UX + Security  
**Severity:** Medium

Access tokens expire in 15 minutes. The `AuthProvider` has no mechanism to detect expiry and silently refresh. In practice, users will get random `401` errors mid-session when their 15-minute token lapses. There is a `/api/auth/refresh` endpoint but nothing calls it automatically.

**Recommendation:**

Implement an Axios/fetch interceptor or a `useEffect`-based token refresh cycle:

```typescript
// Option: proactive refresh before expiry
// Decode token exp claim, schedule refresh at (exp - 60s)
// Or: reactive — intercept 401 responses, call /api/auth/refresh, retry original request
```

A common pattern is to decode the JWT client-side (without verification — just for the `exp` claim), set a `setTimeout` for `(exp * 1000 - Date.now() - 60_000)`, and call refresh before expiry.

#### M3 — `wrangler.toml` Leaks Secret Names with Empty Values
**Category:** Configuration / Secret Management  
**Severity:** Medium

Setting `JWT_SECRET = ""` in `wrangler.toml` is problematic: it trains developers to put real values there (which gets committed to version control), and the empty string fallback means a misconfigured deployment would sign tokens with an empty secret — silently, without error.

**Recommendation:**

Remove secret declarations from `wrangler.toml` entirely. Use only `wrangler secret put`:

```toml
# wrangler.toml — no secrets here
[vars]
ENVIRONMENT = "production"
```

```bash
wrangler secret put JWT_SECRET
wrangler secret put JWT_REFRESH_SECRET
```

Add a startup assertion in `jwtService.ts`:

```typescript
if (!process.env.JWT_SECRET || process.env.JWT_SECRET.length < 32) {
  throw new Error("JWT_SECRET is missing or too short. Refusing to start.");
}
```

#### M4 — `SameSite=Strict` May Break Legitimate Redirect Flows
**Category:** UX / Security Trade-off  
**Severity:** Medium

`SameSite=Strict` means the refresh token cookie is **not sent on any cross-site navigation**, including top-level navigations from external links or email "click to open" flows. If a user follows a link to the app, the cookie won't be present on the first load, forcing a re-login even if their session is valid.

**Recommendation:**

Consider `SameSite=Lax` with explicit CSRF token validation on state-mutating auth endpoints, or document this trade-off explicitly so the team understands why returning users from external links will always land on the login page.

---

#### M5 — Missing `config.matcher` Export Scope Risk
**Category:** Security / Middleware  
**Severity:** Medium

The `middleware.ts` matcher `["/api/bookmarks/:path*"]` does not protect `/api/bookmarks` (without trailing path) — only sub-paths. The `GET /api/bookmarks` (list all) route may be unprotected depending on Next.js matcher semantics.

**Recommendation:**

Validate this explicitly. Update matcher to:

```typescript
export const config = {
  matcher: ["/api/bookmarks", "/api/bookmarks/:path*"],
};
```

Write an integration test that hits `GET /api/bookmarks` without a token and asserts `401`.

#### L2 — No Structured Error Logging

#### L3 — `role` Claim in JWT Is Not Re-validated Against DB
**Severity:** Low | **Category:** Authorization

If a user's role is changed in the database (e.g., a user is demoted from `admin`), their existing access tokens will continue to carry the old role for up to 15 minutes. For the current spec this is acceptable, but it should be a documented known limitation.

#### L4 — Tags Field Serialization Not Defined
**Severity:** Low | **Category:** Data Integrity

The `Bookmark` model shows `tags: string[]` but D1 is SQLite — arrays aren't a native type. The plan mentions "stored as JSON array" but the `bookmarkRepository` is not implemented, leaving serialization/deserialization as an implicit assumption.

Explicitly define `JSON.stringify`/`JSON.parse` round-trips in the repository layer and add a migration comment.
