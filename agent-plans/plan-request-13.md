# Product Specification: PCS App V2

## 1. Summary
PCS App V2 is a modernized application designed to streamline the process of managing PCS (Permanent Change of Station) tasks for users, such as service members and support staff. The app will offer intuitive task management, document uploads, and progress tracking, accessible via a clean web interface.

---

## 2. User Stories

### User Story 1: Task Management
**As a service member,**  
I want to create, update, and mark PCS-related tasks as complete  
**so that** I can stay organized throughout my move.

**Acceptance Criteria:**
- Users can add, edit, and delete PCS tasks.
- Users can set due dates and mark tasks as "completed."
- The app displays a list of all active and completed tasks.

---

### User Story 2: Document Upload
**As a service member,**  
I want to upload and securely store important PCS documents  
**so that** I can easily access them during my relocation.

**Acceptance Criteria:**
- Users can upload PDF/image files for various PCS documentation.
- Files are stored securely and listed with upload dates.
- Users can view or delete previously uploaded documents.

---

### User Story 3: Progress Tracking
**As a user,**  
I want to see an overview of my PCS progress  
**so that** I can quickly understand what's completed and what's pending.

**Acceptance Criteria:**
- The app provides a summary dashboard with task completion status.
- Visual indicators (e.g., progress bars or percentage complete) are shown.

---

## 3. Scope

### In Scope
- User registration and authentication.
- Task CRUD (Create, Read, Update, Delete) operations.
- Secure document upload and management.
- Dashboard with task progress tracking.
- Intuitive and responsive web interface.

### Out of Scope

# PCS App V2 — Technical Architecture

## 1. Overview

PCS App V2 is a full-stack Next.js 14 application using the App Router, deployed on Cloudflare Pages with Workers for API routes. Authentication is handled via Clerk, file storage via Cloudflare R2, and persistence via a Cloudflare D1 (SQLite) database — keeping the entire stack on Cloudflare's edge infrastructure for low latency and minimal operational overhead.

## 4. Concerns

### 🔴 CRITICAL — Missing Row-Level Authorization on Document Access

**Category:** Security / Authorization

**Description:** The presigned URL flow has a critical gap. `GET /api/documents/[id]/url` generates a signed R2 URL for document viewing. The plan does not explicitly state that the server verifies the requesting user **owns** the document before generating the URL. If only authentication (is the user logged in?) is checked but not authorization (does this user own document `[id]`?), any authenticated user can view any other user's PCS documents by guessing or enumerating document IDs. Given that PCS documents include military orders, financial records, and PII, this is a severe exposure.

**Specific Recommendation:**
```typescript
// In GET /api/documents/[id]/url route handler
const { userId } = requireAuth();

// THIS CHECK IS MANDATORY — do not skip it
const doc = await db
  .prepare('SELECT * FROM documents WHERE id = ? AND user_id = ?')
  .bind(documentId, userId)
  .first();

if (!doc) {
  return new Response('Not Found', { status: 404 });
  // Return 404, not 403 — do not confirm the document exists
}
```

The same pattern must apply to `DELETE /api/documents/[id]`, `PATCH /api/tasks/[id]`, and `DELETE /api/tasks/[id]`. Every `[id]`-parameterized route must bind `AND user_id = ?` in its query.

---

### 🔴 HIGH — R2 Storage Key Structure Enables Path Traversal Risk

**Category:** Security / Input Validation

**Description:** The architecture defines the storage key as `userId/docId/filename` where `filename` is the original user-supplied filename. If `filename` is not sanitized, a malicious user could supply a filename like `../../other-user-id/sensitive-doc.pdf`, potentially manipulating the storage key in unexpected ways depending on R2's path resolution behavior. Additionally, the raw filename in the key leaks original document names in any R2 logs or presigned URLs.

**Specific Recommendation:**
```typescript
// In lib/r2.ts or upload-url route
import { nanoid } from 'nanoid';
import path from 'path';

function buildStorageKey(userId: string, docId: string, filename: string): string {
  // Strip to extension only — never trust user-supplied path components
  const ext = path.extname(filename).toLowerCase().replace(/[^a-z0-9.]/g, '');
  const safeExt = ['.pdf', '.jpg', '.jpeg', '.png', '.gif', '.webp'].includes(ext) 
    ? ext 
    : '';
  return `${userId}/${docId}/${nanoid()}${safeExt}`;
}
```

Store the original `filename` only in the D1 `documents` table (for display), never in the R2 key.

---

### 🟠 HIGH — No File Validation at Upload Time

**Category:** Security / Input Validation

**Description:** The plan mentions "PDF/image files" in the spec but does not define where MIME type and file size enforcement occurs. The presigned URL endpoint accepts `fileType` and `fileSize` from the client — both of which are **entirely client-controlled and untrustworthy**. A user can claim their executable is `application/pdf` and upload it. R2 presigned URLs do not automatically enforce content-type restrictions unless explicitly configured.

**Specific Recommendation:**

1. On the `/api/documents/upload-url` route, enforce an allowlist server-side:
```typescript
const ALLOWED_MIME_TYPES = new Set([
  'application/pdf',
  'image/jpeg',
  'image/png',
  'image/gif',
  'image/webp',
]);

const MAX_FILE_SIZE_BYTES = 10 * 1024 * 1024; // 10MB — define this explicitly

if (!ALLOWED_MIME_TYPES.has(fileType)) {
  return new Response('Unsupported file type', { status: 415 });
}
if (fileSize > MAX_FILE_SIZE_BYTES) {
  return new Response('File too large', { status: 413 });
}
```

2. When generating the presigned PUT URL, set the `Content-Type` condition in the presign parameters so R2 rejects uploads that don't match.

3. Note: True server-side MIME magic-byte validation (reading the file header) would require proxying uploads through the Worker, which conflicts with the presigned URL pattern. The above is the practical mitigation. Document this tradeoff explicitly.

---

### 🟠 HIGH — No Rate Limiting Defined

**Category:** Security / Availability

**Description:** The plan has no mention of rate limiting on any API endpoint. This is particularly dangerous for:
- `POST /api/documents/upload-url` — could be hammered to generate thousands of presigned URLs, enabling storage abuse
- `POST /api/tasks` — could be spammed to fill D1 with garbage rows
- Auth endpoints — though Clerk handles its own rate limiting on auth flows

**Specific Recommendation:**

Cloudflare Workers support rate limiting natively via the [Rate Limiting API](https://developers.cloudflare.com/workers/runtime-apis/bindings/rate-limit/). Add it to `wrangler.toml` and apply at minimum to the upload-url and document creation routes:

```toml
[[unsafe.bindings]]
name = "RATE_LIMITER"
type = "ratelimit"
namespace_id = "1"
simple = { limit = 20, period = 60 }
```

Add a per-user rate limit on document uploads (e.g., 20 uploads/minute) and task creation (e.g., 100 tasks/minute).

---

### 🟡 MEDIUM — Presigned URL Expiry Strategy Is Partially Defined

**Category:** Security / Design

**Description:** The plan specifies 15-minute expiry for signed GET URLs (good) but does not define expiry for PUT upload URLs. A PUT URL that lives too long allows delayed uploads to arbitrary storage keys after the upload session should have expired.

**Specific Recommendation:** Explicitly set PUT presigned URL expiry to **5 minutes** in `lib/r2.ts`. Document both values as named constants:

```typescript
const R2_PUT_URL_EXPIRY_SECONDS = 300;   // 5 minutes
const R2_GET_URL_EXPIRY_SECONDS = 900;   // 15 minutes
```

---

### 🟡 MEDIUM — No Orphaned R2 Object Cleanup Strategy

**Category:** Reliability / Data Integrity

**Description:** The two-step upload flow (get presigned URL → upload to R2 → POST metadata to D1) has an inherent race condition: a user can upload a file to R2 via the presigned URL but never call `POST /api/documents` to save metadata. This results in orphaned R2 objects that consume storage without being tracked or deletable by the user.

**Specific Recommendation:** Two mitigations:

1. **Short-term:** In the `POST /api/documents/upload-url` route, save a "pending" record to D1 with the `storage_key` and a `status = 'pending'` field. `POST /api/documents` then updates it to `status = 'confirmed'`.

2. **Long-term:** Add a scheduled Cloudflare Worker cron (one line in `wrangler.toml`) to clean up pending records and their R2 objects older than 1 hour:
```toml
[triggers]
crons = ["0 * * * *"]  # hourly
```

---

### 🟡 MEDIUM — D1 Has No Indexes Defined

**Category:** Performance / Database Design

**Description:** The migrations define tables but no indexes. Every `GET /api/tasks`, `GET /api/documents`, and `GET /api/dashboard` will execute full table scans filtering by `user_id`. While D1 on SQLite handles small datasets fine, this is a bad habit to start with and will degrade predictably as row counts grow.

**Specific Recommendation:** Add to migrations:

```sql
-- 0001_create_tasks.sql addition
CREATE INDEX idx_tasks_user_id ON tasks(user_id);
CREATE INDEX idx_tasks_user_status ON tasks(user_id, status);

-- 0002_create_documents.sql addition
CREATE INDEX idx_documents_user_id ON documents(user_id);
CREATE INDEX idx_documents_user_uploaded ON documents(user_id, uploaded_at DESC);
```

---

### 🟡 MEDIUM — Dashboard Endpoint Design is Vague

**Category:** Design / Performance

**Description:** Step 13 says "single query joining task counts and recent documents into DashboardStats" but D1/SQLite cannot efficiently join aggregate counts and a row-list in a single query without a UNION or subquery structure. This ambiguity will lead to either an inefficient implementation or multiple round trips at implementation time.

**Specific Recommendation:** Define the implementation explicitly now:

```sql
-- Query 1: Aggregates (single row result)
SELECT
  COUNT(*) as total_tasks,
  SUM(CASE WHEN status = 'completed' THEN 1 ELSE 0 END) as completed_tasks
FROM tasks
WHERE user_id = ?;

-- Query 2: Recent documents (separate query)
SELECT id, filename, file_type, file_size, uploaded_at
FROM documents
WHERE user_id = ?
ORDER BY uploaded_at DESC
LIMIT 5;

-- Query 3: Document count
SELECT COUNT(*) as total_documents FROM documents WHERE user_id = ?;
```

Three indexed queries at the edge will be fast. Don't over-engineer a single-query solution here.

---

### 🟢 LOW — `storageKey` Exposed in Document API Response

**Category:** Security / Information Disclosure

**Description:** The `Document` TypeScript interface includes `storageKey`, and if the `GET /api/documents` endpoint returns this field directly to the client, it exposes internal R2 path structure. This is minor but represents unnecessary information disclosure.

**Specific Recommendation:** Create a `PublicDocument` type that omits `storageKey` for client-facing responses. Only use `storageKey` server-side.

```typescript
export type PublicDocument = Omit<Document, 'storageKey'>;
```

---

### 🟢 LOW — No CSP or Security Headers Defined

**Category:** Security / HTTP Headers

**Description:** The plan has no mention of HTTP security headers (Content Security Policy, X-Frame-Options, X-Content-Type-Options). Given that documents are rendered via signed URLs in new tabs and the app handles file uploads, a missing CSP could enable XSS escalation.

**Specific Recommendation:** Add to `next.config.ts`:

```typescript
const securityHeaders = [
  { key: 'X-Content-Type-Options', value: 'nosniff' },
  { key: 'X-Frame-Options', value: 'DENY' },
  { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
  {
    key: 'Content-Security-Policy',
    value: `default-src 'self'; script-src 'self'; img-src 'self' data: blob:; connect-src 'self' *.clerkinc.com *.r2.cloudflarestorage.com;`
  },
];
```

## 5. Security Notes

### Authentication Model
Clerk handles the authentication surface well. The critical gap is that **authentication ≠ authorization**. The plan uses Clerk correctly for "who is this user?" but does not explicitly address "is this user allowed to touch this resource?" See the CRITICAL concern above.


|---|---|

| `DELETE /api/tasks/[id]` | ✅ Required |

### Data Classification
PCS documents (military orders, financial records, housing documents) are **sensitive but unclassified (SBU)** personal data. Cloudflare R2 encryption at rest is on by default, which is good. The plan should explicitly state that:
- No documents are ever returned directly through the Worker (signed URL pattern is correct)
- Signed URLs are single-use scoped and should never be logged
- Clerk user IDs used as D1 partition keys are appropriate (never use PII like email as a key)

### Threat Model Gaps
The plan has no mention of:
- What happens if a Clerk token is compromised (answer: Clerk's session revocation handles this, but worth documenting)
- Whether document metadata (filenames) should be considered sensitive (they likely are — "DD214.pdf" reveals a lot)

### D1 / SQLite Constraints
